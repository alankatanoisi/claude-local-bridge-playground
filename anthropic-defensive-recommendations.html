Detecting & Mitigating Subscription-Client Spoofing and Billing-Class Circumvention

Threat model and recommended controls for distinguishing genuine interactive Claude Code usage from programmatic traffic replaying the official client's credentials and fingerprint.

Prepared for Anthropic's Trust & Safety, anti-abuse, and platform engineering teams. The cover-email author is a subscriber who independently explored this class of workaround; this document is written from the defender's perspective and is intended to help Anthropic get ahead of the technique rather than react to it.

1. Executive summary

A class of user-side tooling captures the credentials and request fingerprint of the official, interactive Claude Code client and replays them from a programmatic agent loop, so that programmatic / Agent-SDK-style usage is metered as interactive subscription usage rather than against the programmatic credit pool.

The root weakness is structural: billing class and client identity are currently inferred from client-supplied, replayable HTTP signals — user-agent, anthropic-beta, x-app, session-id headers, Stainless headers, and an opaque x-anthropic-billing-header / cch value. Any value the client can send, an imitator can resend.

The durable fix is to stop treating soft signals as a trust boundary and move billing/session identity onto cryptographically verifiable, short-lived, sender-constrained claims issued by Anthropic. Soft-signal analysis (fingerprint consistency, replay, version drift) remains valuable as a secondary forensic layer, not as the root control.

Recommendations are tiered — Tier 0 closes the trust boundary, Tier 1 hardens soft signals during migration, Tier 2 adds detection — with explicit guardrails so the large population of legitimate programmatic users (especially sanctioned third-party tools on the new Agent SDK credit) is correctly routed rather than penalized.

2. Threat model

Actor: an authenticated subscriber (consumer OAuth, Pro/Max) — not an account thief. Credentials are the user's own.

Capability: local interception of the official client's outbound requests (e.g., a VS Code extension reading keychain credentials via a local proxy/interceptor), capturing the full header set and replaying it from arbitrary local processes.

Objective: route programmatic agent traffic (custom runners, multi-CLI orchestration, claude -p-style calls wrapped in a bespoke loop) through the request path and headers the server associates with interactive metering.

Why it works today: the server cannot cryptographically distinguish "the official interactive client made this request" from "something replayed the official client's headers." Billing class is a function of spoofable inputs.

What breaks it: any control requiring the client to prove possession of a secret, or to present a server-signed, short-lived, request-bound claim an interceptor cannot mint or meaningfully reuse.

3. Root-cause analysis: the trust boundary is in the wrong place

Signal replayed todayWhy it is not a trust boundary
user-agent (e.g. claude-cli/x.y.z)Free-text, trivially set
anthropic-beta flag set + orderingStatic, copyable, observable
x-app, x-claude-code-session-idClient-asserted strings
Stainless headers (x-stainless-*)Library-emitted, copyable
x-anthropic-billing-header / cch hashOpaque value, captured once and replayed indefinitely
Principle: treat all of the above as untrusted hints. Useful for forensics and UX, never for entitlement decisions. The entitlement decision must rest on something the requester cannot forge.

Tier 0 Move billing/session identity onto verifiable claims (the real fix)

4.1 Server-signed, short-lived session/billing token

Replace the opaque cch with a short-lived JWS/PASETO token issued by Anthropic at interactive-session start and verified server-side on every /v1/messages call. Suggested claims:

  • sub — account / user ID
  • tid — credential ID (binds claim to a specific token)
  • client — client type (interactive CLI, VS Code, programmatic)
  • cc_ver — official client version, server-asserted, not echoed from the client
  • mode — interactive vs. programmatic
  • sid — session ID  ·  seq/nonce — monotonic sequence / per-request nonce
  • exp — short expiry (minutes)  ·  bh — body/request hash where feasible

Signed with a key the client never holds, so a replayed or copied token fails signature / exp / seq / bh checks. Fail closed: absent, stale, or non-verifying → programmatic credit path, not interactive.

4.2 Sender-constrained tokens (proof-of-possession)

Adopt DPoP (RFC 9449) or mTLS-style binding so the access token is bound to a key held only by the genuine client. Each request carries a fresh signed proof over method + URL + timestamp (+ optional body hash). An interceptor that captures headers but not the private key cannot produce valid proofs. This single control neutralizes pure header replay.

4.3 Separate token classes by issuance flow; enforce usage class

Issue distinct credential classes for interactive Claude Code login vs. programmatic/Agent-SDK usage, and enforce the class at the metering boundary. An interactive-issued token used programmatically is metered as programmatic (or rejected), regardless of headers — decoupling entitlement from fingerprint.

4.4 Client attestation at session start

For interactive metering, require lightweight attestation that an official client instance is actually running (control-plane handshake / heartbeat). Bind the interactive lease to a live, attested session. This most directly defeats "headers without a real client behind them."

4.5 Interactive-session lease + policy gate

IF   consumer OAuth
 AND  valid, unexpired, sender-constrained interactive lease
 AND  attested active official client session
THEN meter as interactive subscription usage
ELSE meter as Agent SDK / programmatic credit

The lease is short-lived, renewed by the live client, bound to credential + client + session, non-transferable. Absence of a valid lease is not an error — it is simply the programmatic path.

Tier 1 Harden the soft signals during migration

  • Bind and rotate cch. Derive it from (cc_version, build, credential/session binding, time window); recompute server-side; reject values that are stale, that don't correspond to the claimed cc_version, or that are reused across many sessions. Breaks "capture once, replay forever."
  • Maintain an authoritative version-tuple matrix. Ground-truth of {CLI, Agent SDK, x-stainless-package-version, VS Code extension, runtime, OS/arch, beta-flag set+ordering} that actually shipped together. Flag or reject impossible tuples.
  • Pin and rotate beta-flag sets. Record the exact anthropic-beta combination + ordering per released version. A set matching no shipped version — or lagging past a forced-upgrade window — is high-confidence anomalous. Rotate names/ordering periodically to shorten any captured fingerprint's useful life.
  • Enforce forced-upgrade windows. After a deprecation deadline, refuse interactive metering for clients still claiming superseded versions.

These don't make spoofing impossible; they make captured fingerprints expire quickly — the practical win.

Tier 2 Detection & forensic signals (secondary layer)

Retain as investigation triggers and risk scores — never as the primary boundary:

  • Replay detectors: same header tuple / cch / session-id reused across many accounts, or far more often than human-paced interactive use.
  • Fingerprint-consistency checks: impossible CLI × extension × x-stainless-* × runtime × OS/arch × beta-flag combinations.
  • Version-drift detectors: stale client versions after forced-upgrade windows; old billing-header values after a rotation.
  • Session-existence cross-check: requests claiming claude-vscode while no VS Code control-plane / attestation session exists.
  • OAuth-client consistency: x-app / user-agent claims inconsistent with the OAuth client ID the token was issued to.
  • System-prompt structure fingerprinting: hash the system-block structure; identical hashes across unrelated suspicious accounts is a clustering signal.
  • Behavioral / content parity: compare to genuine interactive Claude Code — count_tokens preflight presence/cadence, tool-use patterns, turn structure, streaming behavior.

Use for risk scoring and human review, not automated punitive action, to keep false positives low.

7. Prioritization

  1. DPoP / sender-constrained tokens (4.2) + token-class enforcement (4.3) — highest leverage; structurally defeat header replay.
  2. Signed session/billing token (4.1) + lease & policy gate (4.5) — formalize the entitlement decision on verifiable state.
  3. Client attestation (4.4) — closes the "headers without a live client" gap.
  4. Tier 1 — deploy immediately as interim mitigation; cheap relative to impact.
  5. Tier 2 — ongoing forensic layer; feeds investigations and tunes the above.

8. False-positive guardrails (do not break legitimate users)

  • Sanctioned programmatic use is not abuse. With the Agent SDK credit covering third-party tools built on the Agent SDK, the goal is correct routing (interactive vs. credit), not blocking programmatic use. Mis-metering legitimate Agent SDK traffic as a violation is a worse outcome than the abuse itself.
  • Fail toward the credit path, not toward errors, when interactive entitlement can't be verified.
  • No automated penalties on soft signals alone. Tier 2 gates review, not bans.
  • Account for legitimate version skew. Honest users lag updates; treat drift as actionable only past explicit forced-upgrade deadlines.
  • Privacy-preserving attestation. Collect no more device data than necessary; document it for users.

9. One-line summary for leadership

Billing class is currently inferred from spoofable client headers; the fix is to issue short-lived, sender-constrained, server-signed session/billing claims and gate interactive metering on a verifiable, attested interactive lease — keeping fingerprint/replay analysis as a secondary forensic layer, with guardrails so sanctioned programmatic (Agent SDK) usage is correctly routed rather than penalized.
