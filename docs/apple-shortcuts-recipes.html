Runtime capabilities these recipes exploit

  • Clean stdout — usage/cost goes to stderr, so Shortcuts gets pure answer text
  • Multimodal reads — read_file attaches images (≤7MB) and PDFs (≤10MB) as model content
  • Permission modes — --plan, --accept-edits, --dont-ask, profiles like review-only
  • Prompt templates — --prompt-template review --prompt-arg focus=… with typed parameters
  • File agents — code-reviewer, debugger, refactoring-specialist, test-automator
  • Worktree isolation — risky edits land on fresh branches, never the main checkout
  • Budgets — --max-cost-usd, --budget-output-tokens hard-stop runaway runs
  • Whole-run rollback — local-bridge-undo.js last-run --yes reverts an entire run
  • Session resume — --continue picks up the latest transcript
  • Throwaway labs — create-runner-throwaway-lab.js mints disposable sandboxes
01

📸 Screenshot → Instant Code Review

Share any screenshot of code, an error dialog, or a UI bug straight into a multimodal review.

Share Sheet · Images read-only multimodal --bare

macOS screenshots land in the Share Sheet; this shortcut saves the shared image into the repo's temp inbox, then lets the runner see it via multimodal read_file and diagnose what's on screen — stack traces, Xcode errors, broken layouts, whiteboard photos of architecture diagrams.

Shortcut actions
  1. Receive Images from Share Sheet, Quick Actions
  2. Save File → ~/Developer/claude-local-bridge-playground/.bridge-runner/inbox/shot.png (overwrite)
  3. Run Shell Script (zsh, pass input: as arguments)
  4. Stop and output Shell Script Result → fallback: Show Result
cd "/Users/alanman/Developer/claude-local-bridge-playground"
mkdir -p .bridge-runner/inbox
node bin/local-bridge-runner.js --bare --dont-ask --max-cost-usd 0.25 \
  "Read the image at .bridge-runner/inbox/shot.png. Identify what is on screen
   (code, error, UI). Explain the likely root cause and give a concrete fix.
   If it is code, review it for bugs." 2>/dev/null
Why it's clever: zero typing — screenshot ⌘⇧4, hit Share, and the answer pops up as a Quick Look sheet. The 2>/dev/null keeps the stderr usage line out of the result.

🎙️ Siri "Plan This Feature" — Voice to Engineering Plan

Dictate a feature idea to Siri; get back a structured, repo-aware implementation plan in Notes.

Siri · Voice --plan mode repo-context

Say "Hey Siri, plan this feature." Siri asks what you want, and the runner explores the repo in plan mode — it describes what it would do without executing a single write. The plan is appended to an Apple Note so ideas captured on a walk survive until you're back at the keyboard.

Shortcut actions
  1. Ask for Input (Text): "What should I plan?" — dictation-friendly
  2. Run Shell Script with Provided Input as $1 (pass: as arguments)
  3. Append to Note "Runner Plans" ← Shell Script Result
  4. Show Notification "Plan captured ✅"
cd "/Users/alanman/Developer/claude-local-bridge-playground"
node bin/local-bridge-runner.js --plan --include-repo-context --include-repo-map \
  --dont-ask --max-cost-usd 0.50 \
  "Draft a step-by-step implementation plan for: $1.
   List the files involved, risks, and a test strategy. Do not edit anything." 2>/dev/null
Why it's clever: --plan makes voice-triggered agent runs fundamentally safe — the model gets read tools and repo maps but every mutation is described, not performed.

📋 Clipboard Refactor Surgeon

Copy ugly code anywhere → hotkey → refactored code lands back on your clipboard.

Keyboard hotkey / Menu bar writes to sandbox only file agent

Uses the shipped refactoring-specialist agent from .bridge-runner/agents/. The clipboard is written into a scratch file inside a sandbox folder, the agent refactors it with --accept-edits scoped to that sandbox, and the improved file is read back onto the clipboard. Works from any app — Slack, browser, Mail.

Shortcut actions
  1. Get Clipboard
  2. Run Shell Script with Clipboard as stdin (pass input: to stdin)
  3. Copy to Clipboard ← Shell Script Result
  4. Show Notification "Refactored ✂️ → clipboard"
REPO="/Users/alanman/Developer/claude-local-bridge-playground"
SANDBOX="$HOME/Documents/runner-clipboard-sandbox"
mkdir -p "$SANDBOX" && cat > "$SANDBOX/snippet.txt"   # clipboard → file
cd "$REPO"
node bin/local-bridge-runner.js --cwd "$SANDBOX" --agent refactoring-specialist \
  --accept-edits --dont-ask --max-cost-usd 0.40 \
  "Refactor snippet.txt in place: improve naming, structure, and clarity while
   preserving behavior. Keep the same language. Save the result to snippet.txt." \
  1>/dev/null 2>&1
cat "$SANDBOX/snippet.txt"   # file → clipboard
Why it's clever: --cwd jails all writes to a scratch folder, so --accept-edits is safe even on a hotkey. The runner never touches a real repo.
04

📂 Finder Quick Action: "Grill This Folder"

Right-click any project folder → adversarial findings-first code review, no edits possible.

Finder Quick Action · Folders review-only profile prompt template

Registers as a Finder Quick Action that receives folders. The selected folder path arrives as $1 and becomes the runner's --cwd. The built-in grill prompt template plus the review-only capability profile means the run is aggressive in analysis and physically incapable of editing.

Shortcut actions
  1. Receive Files/Folders from Quick Actions in Finder
  2. Run Shell Script (pass input: as arguments — folder path is $1)
  3. Save File Shell Script Result → ~/Desktop/review-report.txt, then Open File
cd "/Users/alanman/Developer/claude-local-bridge-playground"
node bin/local-bridge-runner.js --cwd "$1" --profile review-only \
  --prompt-template grill --dont-ask --max-cost-usd 1.00 \
  --human-log "$HOME/Desktop/review-$(date +%H%M%S).log" \
  "Review this project. Lead with the 5 most severe concrete findings,
   each with file:line references and a suggested fix." 2>/dev/null
Why it's clever: profiles + templates compose — the same shortcut can review any repo on disk because the runner is repo-agnostic and takes --cwd from the Finder selection.

📄 PDF Interrogator — Ask Questions of Any Document

Share a PDF (spec, invoice, paper, contract) and interrogate it conversationally.

Share Sheet · PDFs read-only multimodal PDF --continue follow-ups

The runner's read_file attaches PDFs up to 10MB as native Anthropic content blocks. This shortcut drops the shared PDF into the inbox, asks you for a question via a dialog, and answers from the document. A companion one-action shortcut using --continue resumes the same transcript for follow-up questions.

Shortcut actions
  1. Receive PDFs from Share Sheet
  2. Save File → …/.bridge-runner/inbox/doc.pdf (overwrite)
  3. Ask for Input: "What do you want to know?" → $1
  4. Run Shell Script → Show Result
cd "/Users/alanman/Developer/claude-local-bridge-playground"
node bin/local-bridge-runner.js --bare --dont-ask --max-cost-usd 0.60 \
  "Read .bridge-runner/inbox/doc.pdf and answer precisely,
   citing page/section where possible: $1" 2>/dev/null

# Follow-up shortcut ("Ask the PDF again"), one action:
node bin/local-bridge-runner.js --continue --dont-ask "$1" 2>/dev/null
Why it's clever: --continue resumes the latest transcript from ~/.bridge-runner/logs/, turning two stateless shortcuts into a stateful document-chat pair.

🌳 Worktree Experiment Launcher

Fire off a risky refactor idea onto an isolated git branch — your checkout is never touched.

Menu bar / Ask for Input edits on isolated branch enter_worktree

The runner's enter_worktree tool creates a fresh branch under ~/.bridge-runner/worktrees/ and moves all subsequent tool calls into it. This shortcut asks for an experiment idea, tells the agent to do all work inside a worktree, and reports the branch name back — so you review with git diff main...branch at your leisure.

Shortcut actions
  1. Ask for Input: "Describe the experiment" → $1
  2. Run Shell Script
  3. Show Result (branch name + summary of changes)
cd "/Users/alanman/Developer/claude-local-bridge-playground"
node bin/local-bridge-runner.js --accept-edits --dont-ask --max-cost-usd 1.50 \
  --max-steps 30 \
  "First call enter_worktree with a branch suffix describing this task.
   Then, entirely inside the worktree: $1.
   Do NOT call exit_worktree with cleanup — preserve the work.
   Finish by reporting the worktree branch name and a bullet summary of every
   file you changed." 2>/dev/null
Why it's clever: --accept-edits feels dangerous from a Shortcut, but worktree isolation means the blast radius is a disposable branch. Cleanup later with node bin/local-bridge-runner.js runner worktrees list.

🌙 Nightly Test Sentinel (Automation)

A scheduled 2 AM automation that runs the test suite, has the agent triage failures, and leaves a morning report.

Automation · Time of day --allow-shell test-automator agent budget capped

Shortcuts personal automations can run on a schedule with "Run Immediately". This one lets the test-automator file agent run npm test via --allow-shell, triage any failures down to root cause, and write a dated report. A notification tells you PASS/FAIL before you've had coffee.

Shortcut actions
  1. Automation: Time of Day 2:00 AM, Run Immediately
  2. Run Shell Script
  3. Show Notification ← first line of Shell Script Result
cd "/Users/alanman/Developer/claude-local-bridge-playground"
REPORT="$HOME/Desktop/nightly-sentinel-$(date +%Y-%m-%d).md"
node bin/local-bridge-runner.js --agent test-automator --allow-shell --dont-ask \
  --max-cost-usd 2.00 --budget-output-tokens 20000 --max-steps 40 \
  --human-log "$REPORT" \
  "Run 'npm test'. If everything passes, output exactly 'PASS: all tests green'
   plus a one-line stat. If anything fails, diagnose each failure to root cause
   (read the code, don't guess), propose fixes, but DO NOT edit files.
   Start your final answer with PASS: or FAIL:." 2>/dev/null
Why it's clever: hard budgets (--max-cost-usd, token caps) make unattended runs safe from runaway loops, and --human-log gives you a full morning-readable transcript regardless of what the notification says.

🧪 Throwaway-Lab Chaos Bench

One tap mints a disposable sandbox repo and stress-tests the agent end-to-end — perfect demo & regression ritual.

Menu bar / Dock full autonomy, sandboxed bench agent

Wraps the repo's own create-runner-throwaway-lab.js helper: it mints a timestamped lab under ~/Documents/…-throwaway-labs/, then unleashes the bench agent with full permissions inside that lab only. Great as a "is my whole stack healthy?" smoke test — bridge, OAuth, tools, shell, edits — from a single Dock icon.

Shortcut actions
  1. Run Shell Script
  2. Show Result (final agent summary + lab path)
cd "/Users/alanman/Developer/claude-local-bridge-playground"
LAB="$(node scripts/create-runner-throwaway-lab.js)"
node bin/local-bridge-runner.js --cwd "$LAB" --agent bench \
  --trust-workspace --allow-shell --accept-edits --dont-ask --chaos-ok \
  --max-steps 24 --max-cost-usd 1.00 \
  "Fix the calculator bug, run npm test, and summarize the result." 2>/dev/null
echo "— lab: $LAB"
Why it's clever: maximum-autonomy flags are fine here because every run gets a fresh sandbox — no stale state, nothing real at risk. It's the repo's own recommended smoke ritual, one tap away.

🕰️ "What Did the Agent Do?" — Archive Historian

Summon a plain-English audit of recent agent runs, with a one-tap whole-run rollback option.

Menu bar · Choose from Menu read-only (audit) optional rollback

Combines two runner CLIs. First, local-bridge-archive.js list and local-bridge-undo.js list-runs pull the raw history; then the runner itself summarizes it in plain English. A Shortcuts Choose from Menu adds a second branch: revert the most recent run using the per-run manifest, with --dry-run shown first.

Shortcut actions
  1. Choose from Menu: "Audit recent runs" / "Preview rollback" / "Revert last run"
  2. Each branch → Run Shell Script (below) → Show Result
# Branch A — audit:
cd "/Users/alanman/Developer/claude-local-bridge-playground"
HIST="$(node bin/local-bridge-undo.js list-runs --cwd . 2>&1 | tail -40)"
node bin/local-bridge-runner.js --bare --dont-ask --max-cost-usd 0.20 \
  "Here is my agent run history:
$HIST
Summarize in plain English: what ran recently, what was edited, anything unusual." 2>/dev/null

# Branch B — preview rollback (changes nothing):
node bin/local-bridge-undo.js last-run --cwd . --dry-run

# Branch C — actually revert the whole last run:
node bin/local-bridge-undo.js last-run --cwd . --yes
Why it's clever: the manifest system means an entire multi-file agent run is one reversible unit — diverged files are skipped automatically, so newer manual work is never clobbered from a Shortcut.

🐝 Parallel Swarm Review — spawn_agent Fan-Out

One shortcut, three specialist child agents in parallel: correctness, security, and test coverage.

Finder Quick Action / Hotkey read-only children spawn_agent orchestration

The top-level runner acts as an orchestrator: it delegates three focused subtasks to child agents via spawn_agent, each with its own isolated context window, then merges their findings into a single ranked report. Point it at any repo from Finder. Because children can't spawn further children and everything is look-only, it's a safe heavyweight.

Shortcut actions
  1. Receive Folders from Quick Actions in Finder (path = $1)
  2. Run Shell Script
  3. Make Rich Text from Markdown ← result → Quick Look
cd "/Users/alanman/Developer/claude-local-bridge-playground"
node bin/local-bridge-runner.js --cwd "${1:-.}" --profile review-only \
  --dont-ask --max-cost-usd 2.50 --max-steps 20 \
  "You are an orchestrator. Spawn three child agents with spawn_agent, each
   focused on this project:
   1) agent 'code-reviewer' — correctness and logic bugs;
   2) agent 'explore' — security smells, secret handling, unsafe shell use;
   3) agent 'test-automator' — test coverage gaps (analysis only, no shell).
   Merge their reports into one ranked list of findings in Markdown, deduplicated,
   most severe first, with file references." 2>/dev/null
Why it's clever: this is the runner's multi-agent capability made tangible from a right-click — three isolated context windows means each specialist reads deeply without exhausting one shared context, and --dont-ask lets spawns proceed unattended (children inherit the parent's remaining budget).

🛠️ Shared implementation notes

  • All recipes mirror your example shortcut: Receive → Run Shell Script (zsh) → Stop and output. Set "Pass Input" to as arguments when the recipe uses $1, or to stdin for the clipboard recipe.
  • Shortcuts' shell environment is minimal — if node isn't found, use an absolute path (e.g. /opt/homebrew/bin/node) or prepend export PATH="/opt/homebrew/bin:$PATH".
  • The VS Code bridge must be running on localhost:11437; on a 401, run claude /login to refresh the OAuth token.
  • 2>/dev/null hides the stderr usage/cost line from Shortcut output; drop it while debugging, or add --trace-level summary for a local audit trail.
  • Non-interactive runs should always pair --dont-ask with a budget: --max-cost-usd, --max-steps, and token caps. ask_user_question fails closed without a TTY, which is exactly what you want inside Shortcuts.
  • Run any of these from Terminal too: shortcuts run "<name>" -i <input-file>.
