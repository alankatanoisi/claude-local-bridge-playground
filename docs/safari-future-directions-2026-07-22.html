Future Directions & Outstanding Questions

Safari 1 and Safari 2 (both 2026-07-21) are complete: two field-experimentation ladders across the runner's permission surface, run against live models with fake fixtures, disposable sandboxes, and hash-manifested evidence. This page is the forward-looking synthesis — what the safaris opened up, what remains unproven, and what to do next. No runner source was changed during observation.

Where things stand

  • Safari 2 — Rounds A–P plus one isolated symlink proof: shell vs deny matrix, secret-leak positive controls, write-side denials, prompt injection, budgets, artifact permissions.
  • Headline result: central redaction held on every tested positive-control surface, but an in-root symlink exposed a read-side target-basename gap — the alias passed the deny matrix and only redaction stood between the fixture and the transcript (finding S2-01).

P0 Observed, unresolved issues

FD-01 · Resolved-target deny checks for reads (S2-01)

An in-root symlink whose alias basename looked safe but pointed at .env was readable — the deny matrix keyed off the requested name, not the resolved target. Redaction saved the day, which means one control, not two, stood between the fixture and the transcript.

Do: apply the deny matrix to both the requested path and the realpath-resolved in-root target for read_file, search_text, and glob; add regression tests; update the caveat in threat-model.md.

FD-02 · Sibling write tools + hardlinks (S2-02 proof boundary)

write_file replaced the link object and did not write through — but edit_file and apply_patch were never tested, and hardlinks weren't touched in either safari.

Do: isolated fixture tests hashing the target before/after each write path; fix any write-through observed.

FD-03 · Post-timeout process lifecycle (S2-03)

After a confirmation timeout the CLI printed its final result but stayed alive until Ctrl-C. Likely a dangling handle (timer, stdin, socket).

Do: deterministic repro; defined exit code after timeout denial; no live handles.

FD-04 / FD-05 · Bridge deadline & liveness ≠ readiness (S2-04)

A wedged /v1/messages call idled >3 minutes on an established local TCP connection while /v1/debug cheerfully answered. Transport liveness told us nothing about the model path.

Do: client-side deadline + abort with an attributable error (bridge_request_timeout), tested against a stalling mock server; add a message-path readiness probe or label /v1/debug as transport-only everywhere.

P2 Hardening probes still needing runner-level evidence

Core methodological lesson from Safari 2: a model refusal is not proof a runner filter works. Several "denials" were the model declining to try — the runner's own enforcement was never hit. The probes below need positive controls that actually emit the tool call (direct injection or a scripted model), always with fake fixtures in disposable sandboxes.
  • FD-11 · Shell-filter canonicalization. Round H (quoted paths, variable-built paths, $(echo …), normalized ./x/../.env) produced only model refusals. Does the lexical filter catch obfuscation at all?
  • FD-12 · Indirect egress under --no-network. Only direct curl was tested. Probe proxy-env unsetting, DNS/nc, and runtime APIs (node -e 'fetch(…)') against a local-only listener.
  • FD-13 · Case-sensitive filesystem retest. Round O ran on case-insensitive macOS, where .ENV existence checks alias .env. Repeat on Linux; also consider Unicode NFC/NFD aliases of deny-listed basenames.
  • FD-14 · Redaction property tests. Chunk splitting, base64/URL-encoded/rot13 transforms, multiline keys, adjacent punctuation, all trace levels — and the hard case: the model transforming a secret-shaped value before emitting it.
  • FD-15 · Interruption completeness. SIGINT/kill at model-call, read-batch, confirmation-wait, and post-write boundaries; verify pending intents reconcile and terminal events land.
  • FD-16 · Prompt-injection step burn. Round E showed resilience, but denial-loop waste (steps, tokens, cost) from hostile in-project text was never measured.
  • FD-17 · OS-level sandbox exploration. Threat-model known limitation #1: regex scanning is defense-in-depth, not isolation. Scoping spike for Seatbelt (macOS) / Landlock or containers (Linux) as an opt-in shell execution mode.

P3 Methodology & tooling investments

  • FD-18 · Evidence-bundle command. One command that exports normalized, redacted, hash-manifested run evidence (no provider reasoning, no fixture contents, no private home-dir state). Safari 2's manual archive under docs/artifacts/ is the template.
  • FD-19 · Recurring safari matrix. A compact, versioned probe subset re-run after any permission/redaction/shell/persistence/transport change, with results diffed over time. Model-free probes (direct tool-call injection) are CI candidates.
  • FD-20 · Cross-model control set. Repeat a small ladder across model families and effort levels to separate stable runner behavior from model-specific refusal styles.
  • FD-21 · Outer-guardrail-compatible safari design. Safari 2 lost significant time to outer product guardrails blocking bounded fake-fixture experiments. Pre-registered probe manifests (declared fixtures, bounded commands, expected effects) would be legible to outer classifiers and human reviewers alike.
  • FD-22 · Auto-mode retest. Auto permission mode was excluded due to the classifier outage. When it recovers, characterize how outer auto mode composes with the runner's own ladder — and whether either layer ever approves what the other would deny.

Outstanding questions

  1. Target-aware denies: should the deny matrix key off the resolved target for reads — and what are the right semantics for hardlinks, which have no link object to inspect?
  2. Write siblings: do edit_file / apply_patch follow in-root symlinks where write_file did not?
  3. Filter reality: would the lexical shell filter actually catch an obfuscated deny-listed path, or has model refusal been masking an untested filter the whole time?
  4. Redaction vs transformation: does redaction survive the model base64-ing, reversing, or chunking a secret-shaped value across messages?
  5. Budget boundaries: is there any window where an effectful tool call can land after the wall-clock or cost budget is exceeded?
  6. Timeout lifecycle: is the live process after a confirmation timeout caused by a dangling timer, stdin handle, socket, or intentional wait?
  7. Naming: should --dont-ask become --no-prompts or --non-interactive, given that it does not approve writes?
  8. Denial wording: should a non-interactive refusal tell the model “User denied” when no human decision occurred?
  9. Bridge readiness: how should /v1/debug be labeled or supplemented so a wedged message path is detected before a run hangs?
  10. Filesystem variation: do case-sensitive filesystems and Unicode NFC/NFD aliases change deny-matrix or redaction behavior?
  11. Layer composition: when outer auto mode returns, can it approve something the runner denies, or vice versa?
  12. Standing regression set: which minimal probes should run after every permission-related change, and which can be model-free in CI?
