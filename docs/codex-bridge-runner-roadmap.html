Codex Bridge Runner — Feasibility & Roadmap

Can the cc bridge runner architecture be reused to build a Codex local bridge runner that drives OpenAI's Codex models through the same local agent loop? Short answer: yes — this document reviews the current runtime capabilities, measures the provider coupling, and lays out a phased build plan.

TL;DR verdict

✅ Feasible — roughly 85% of the runner carries over as-is

The runner was built with a clean seam between “talk to the model” and “everything else” (tools, safety, permissions, sessions, archives, prompts, profiles, hooks, undo, coordinator). The “everything else” layer never looks at Anthropic-specific data shapes — it consumes a neutral { name, input, id } tool call and produces plain text results. Only four core modules speak the Anthropic Messages dialect natively, and each has a well-understood OpenAI Responses-API equivalent.

The recommended build is a provider adapter inside the runner plus a small standalone Codex credential bridge (a separate local daemon on its own port — the existing Claude bridge is not touched, preserving its transport invariants). Estimated effort: 6 phases, with a usable read-only Codex runner at the end of Phase 4.

⚠️ Two caveats before anything is built.

1. Policy risk (same class as the Claude bridge). A Codex bridge would reuse ChatGPT OAuth credentials created by the official Codex CLI outside that official client. Like the Claude bridge, this is personal research in a gray zone — it is not proof of OpenAI approval, it may break at any time, and it carries some account risk. This must be documented plainly wherever the Codex transport is described.

2. Wire-format facts need re-verification. The Codex CLI and its backend endpoints evolve quickly. Every protocol detail in Part 3 is best current knowledge and is re-verified as the very first roadmap task (Phase 0) against the open-source Codex CLI code before any adapter code is written.

Part 2 — Where the Anthropic-specific code lives

A grep for Anthropic message-shape vocabulary (tool_use, tool_result, content_block, input_schema, cache_control, thinking) finds 121 hits across 39 runner files — but the distribution is very lopsided. Most files have exactly one hit: the input_schema key in their tool definition, which is a mechanical one-line concern. The real coupling is concentrated in four modules:

ModuleAnthropic-specific behaviorCodex-side equivalentPort difficulty
src/runner/model-client.js (262 lines) Messages API request/response; SSE event grammar (message_start, content_block_delta, thinking_delta/signature_delta, input_json_delta) Responses API request/response; SSE grammar (response.output_text.delta, response.function_call_arguments.delta, response.completed) Medium — write a sibling client, same size
src/runner/run.js (1,370 lines) Content-block extraction; cache_control breakpoint budgeting (4-marker budget with bridge reserve); thinking-block + signature preservation across turns; usage field names No manual cache markers (OpenAI caching is automatic); reasoning items with encrypted_content must be replayed — structurally the same “preserve opaque block” problem the runner already solves for thinking signatures Medium — mostly skipping logic per provider, not adding it
src/runner/tool-pipeline.js (532 lines) Assembles tool_use / tool_result message pairs and ids Assembles function_call / function_call_output items keyed by call_id Low — if the adapter translates at the client boundary, this file does not change at all
src/runner/context-compactor.js (337 lines) Compaction ladder walks Anthropic message/content-block shapes Same, if translation happens at the boundary Low — unchanged under the boundary-adapter design
Tool definitions (24 files under src/runner/tools/) input_schema key (Anthropic name for the JSON Schema) parameters key (OpenAI name for the same JSON Schema), optional strict: true Low — one shared mapping function in the adapter; tool files untouched
Key architectural insight.

Because tool-pipeline.js, context-compactor.js, transcripts, the archive, and the golden-eval harness all speak the internal Anthropic-block dialect, the cheapest correct design is to keep that dialect as the runner's internal lingua franca and translate only at the model-client boundary: Codex function_call items become internal tool_use blocks on the way in, and internal tool_result blocks become function_call_output items on the way out. One adapter file, everything downstream unchanged.

Part 3 — What Codex looks like on the wire

Codex CLI
OpenAI's open-source local coding agent (github.com/openai/codex, Apache-2.0, written in Rust). It is the OpenAI counterpart of Claude Code.
Responses API
OpenAI's current model API. Where Anthropic sends a messages array plus a system prompt, OpenAI sends an input item list plus an instructions string.
ChatGPT OAuth
Signing in to Codex CLI with a ChatGPT Plus/Pro account instead of an API key. Tokens are stored locally in ~/.codex/auth.json (access token, refresh token, account id) — the direct analog of Claude Code's keychain entry / ~/.claude/.credentials.json.
SSE (server-sent events)
The streaming format both providers use: a long-lived HTTP response that emits small JSON frames as the model generates.

The structural takeaway: every Anthropic concept the runner relies on has a direct Codex counterpart, and the two hardest problems the runner already solved — streamed partial tool-call JSON assembly and opaque reasoning-block round-tripping — are the same problems in slightly different clothes. Nothing on the Codex side requires a capability the runner lacks; one thing (manual cache budgeting) simply disappears.

Part 4 — Architecture decision: three ways to build it

The playground has a hard transport invariant: the Claude bridge stays Anthropic-native (no /v1/chat/completions, no OpenAI routes, no API-key fallback). So “add OpenAI support to the existing bridge” is off the table by design. That leaves three shapes:

Option A — Standalone Codex credential daemon + runner provider adapter Recommended

A new, small, separate local daemon (a plain Node script — no VS Code extension needed, since Codex credentials live in a readable file rather than behind editor interception) listens on its own port (e.g. localhost:11438), discovers ChatGPT OAuth from ~/.codex/auth.json / env var, handles token refresh, and forwards native Responses-API requests. The runner gains a --provider codex flag that swaps in a Codex model client whose adapter translates to the internal Anthropic-block dialect at the boundary.

  • Pros: preserves the bridge/runner separation the whole repo is built on; credentials stay out of the runner process; keeps the trace-correlation, caller-auth, and redaction patterns; Claude bridge invariants untouched; the daemon is reusable by other local clients later.
  • Cons: one more process to start; small new codebase to maintain (~300–500 lines).

Option B — Provider adapter only, runner talks to OpenAI directly

  • Pros: no new daemon; fastest to a demo.
  • Cons: moves credential discovery, refresh, and redaction into the runner, breaking the “bridge owns auth, runner owns the loop” boundary; loses trace correlation and caller auth; makes the runner itself hold live account tokens, which the threat model currently avoids.

Option C — Fork the repo into a separate codex-local-bridge project

  • Pros: zero risk to the Claude runner; total freedom.
  • Cons: permanent double maintenance of ~10,000 lines of runner code; every safety fix must be applied twice; directly contradicts the “one runner, small extension points” direction of this lab.
Recommendation: Option A.

It is the only shape where the Codex work strengthens rather than dilutes the existing architecture: the provider seam it introduces (Phase 1) is a refactor the runner benefits from even if the Codex daemon is never finished, and the daemon reuses the redaction/caller-auth/trace patterns already proven in the Claude bridge.

Part 5 — Phased build roadmap

Phases are ordered so that each one lands independently, keeps npm test green for the Claude path, and produces something reviewable. “Session” means one focused agent working session. Exit criteria are deliberately test-shaped because the golden-eval harness lets most of this be verified with no live credentials at all.

Phase 0 — Protocol spike & policy write-up 1–2 sessions

new: docs/lab-notes/codex-protocol-notes.md · touches: nothing in src/
  • Read the open-source Codex CLI source and pin: exact endpoint URLs per auth mode, auth.json schema, refresh flow, required headers, request/response JSON, SSE event list.
  • Capture one real Codex CLI request/response pair locally (Codex CLI has debug logging) and store a redacted transcript as the reference fixture.
  • Write the policy-risk section: ChatGPT OAuth reuse outside official clients is a ToS gray zone; personal research framing; account-risk statement mirroring the Claude bridge's.
  • Decide the model list and pricing rows to add to model-pricing.js.

Exit criteria: every row of the Part 3 table above is confirmed or corrected, with a citation into Codex CLI source; go/no-go checkpoint with Alan.

Phase 1 — Provider seam in the runner (no behavior change) 2–3 sessions

touches: src/runner/model-client.js, run.js · new: src/runner/providers/ (interface + anthropic provider) · tests: golden evals must pass unchanged
  • Define a small provider interface: buildRequest(system, tools, messages, opts), send / sendStream, normalizeResponse → internal content blocks, normalizeUsage, cacheStrategy.
  • Move today's Anthropic logic (including applyCacheControlBudget and thinking/signature handling) behind that interface as the default provider. run.js, tool-pipeline.js, and the compactor keep consuming internal blocks and notice nothing.
  • Record a provider field in transcripts, archive turns, and traces (schema-additive only).

Exit criteria: full test suite and all golden evals pass with zero expect-block changes; this proves the seam is behavior-neutral.

Phase 2 — Codex credential daemon (codex-local-bridge) 2–3 sessions

new: bin/codex-local-bridge.js (or a sibling folder), test/codex-bridge/*.test.js · touches: nothing in src/runner or the Claude bridge
  • Plain Node HTTP daemon on localhost:11438: discovers credentials (env var → ~/.codex/auth.json), injects Bearer + account-id, forwards native Responses requests, streams SSE back.
  • Token refresh on 401 (mirror the Claude bridge's “clear cache, retry once” pattern), never logging token bodies; extend secret redaction patterns to OpenAI/ChatGPT token shapes.
  • Port over: optional caller auth, locked debug endpoint, redacted request logging, trace correlation headers.
  • Invariants, mirrored from the Claude side: OAuth-only (no OPENAI_API_KEY upstream fallback in this lab), placeholder client keys never forwarded, debug surfaces redacted.

Exit criteria: curl through the daemon completes a one-shot Responses call and a streamed call; unit tests cover credential priority, refresh, and redaction with fixture tokens.

Phase 3 — Codex provider adapter 3–4 sessions

new: src/runner/providers/codex.js, test/runner/codex-provider.test.js · touches: model-pricing.js
  • Request builder: internal system prompt → instructions; tool definitions → { type: "function", …, parameters }; internal messages/blocks → input items; --effort → reasoning.effort; store: false + reasoning include for stateless multi-turn.
  • Response normalizer: function_call items (JSON-string arguments) → internal tool_use blocks; reasoning items preserved as opaque replayable blocks (reusing the thinking-signature pattern); output text → text blocks.
  • SSE mapper: Responses event grammar → the internal streamed-block assembly postStream already implements (partial tool-arg buffering carries over almost verbatim).
  • Usage normalizer: cached_tokens → cache_read_input_tokens, cache-write counter fixed at 0; add Codex model rows to the pricing table; cacheStrategy: none (skip all cache_control markers).

Exit criteria: unit tests translate the Phase 0 captured fixtures both directions byte-for-byte; a golden-eval case runs the full loop against a scripted fake Codex client.

Phase 4 — Runner integration & first live runs 2–3 sessions

touches: bin/local-bridge-runner.js, src/runner/run.js (flag plumbing only) · docs: runner-quickstart.html, command-builder.html
  • Add --provider anthropic|codex (default anthropic), defaulting --bridge-url to the matching daemon port; env var BRIDGE_RUNNER_PROVIDER.
  • Verify every capability group live in a throwaway lab, in this order: read-only run → --plan → write tools with confirmation → --accept-edits → --allow-shell → spawn_agent → worktrees → --stream.
  • Confirm budgets, cost summary, transcripts, archive, human log, undo manifests, and traces all render correctly with the provider field.
  • Update the command builder with a provider selector and a Codex preset.

Exit criteria: the README throwaway-lab smoke task (“fix the calculator bug, run npm test, summarize”) passes end-to-end on --provider codex.

Phase 5 — Parity hardening & docs 2–3 sessions

touches: docs/threat-model.md, README.md, golden cases, safety.js patterns
  • Golden-eval grammar extension so canned scripts can be authored in either provider dialect; add Codex cases for permission denials, budget stops, repeat-tool detection, and compaction.
  • Close the known gaps found in Phase 4: multimodal (input_image; PDF handling if unsupported, degrade to text extraction with a clear tool note), token-count preflight differences, retry/backoff tuning for the ChatGPT backend.
  • Threat-model update: new credential class (ChatGPT OAuth), new daemon surface, redaction coverage, and the policy-risk statement from Phase 0.
  • Bench: rerun turn-latency and cache-reuse benches; document that Codex cache reuse is automatic and how the usage line reports it.

Exit criteria: npm test, npm run lint, npm run check:docs, golden evals all green; docs updated; a written parity table of anything still Claude-only.

Phase 6 — Optional: dual-provider experiments exploratory

  • Coordinator phases pinned to different providers (e.g. Claude plans, Codex implements, Claude reviews — or the reverse).
  • Cross-provider A/B on the Harbor smoke tasks with the same prompts, comparing cost, steps, and stop reasons from the archive.
  • spawn_agent with a per-child provider override.

Exit criteria: none — this is the payoff playground once Phases 0–5 are stable.

Part 7 — Risks, policy, and invariants

RiskSeverityMitigation
Policy / ToS gray zone. Reusing ChatGPT OAuth outside the official Codex CLI is not an approved integration; OpenAI could block it or action the account. High Same posture as the Claude bridge: personal research only, documented plainly in README/threat model, OAuth-only with no credential capture-and-replay side channels, easy to stop using. Go/no-go checkpoint at end of Phase 0.
Protocol drift. The ChatGPT backend Codex endpoint is not a stable public contract; Codex CLI releases change it. Medium Phase 0 fixtures pin the contract; adapter unit tests fail loudly on shape changes; keep the daemon thin so updates are small.
Token handling. A second live credential class (ChatGPT access + refresh tokens) enters local tooling. Medium Tokens live only in the daemon; redaction patterns extended before first live run; traces/logs/transcripts stay redacted by default; deny matrix already blocks ~/.codex-style credential files from tool reads (verify and extend in Phase 2).
Scope creep into the Claude bridge. Medium Hard rule carried from CLAUDE.md: no OpenAI routes, models, or keys touch src/server.js/src/proxy.js/src/credentials.js. The Codex daemon is a separate entrypoint on a separate port.
Refactor regression risk in Phase 1. run.js is 1,370 lines and load-bearing. Medium Phase 1 ships with zero golden-eval expect changes as its acceptance test; the seam is extraction, not rewrite.
Behavioral differences. Codex models may call tools more/less aggressively, changing loop dynamics (repeat detection thresholds, failure escalation). Low All loop-health knobs are already configurable; archive + autopsy give the data to tune them; Phase 6 A/B makes it measurable.
