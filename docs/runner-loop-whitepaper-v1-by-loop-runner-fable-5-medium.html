The Claude Local Bridge Runner:
An Architectural Analysis of a Local Agentic Execution Loop

Technical White Paper

Claude Local Bridge Playground — Runner Layer · Generated from source inspection of src/runner/run.js, src/runner/tool-pipeline.js, src/runner/kernel/, and CONTEXT.md

Abstract

We present an architectural analysis of the Claude Local Bridge Runner, a local coding-agent harness implemented in Node.js that drives an iterative prompt → model → tool-use → tool-result loop against an Anthropic-compatible /v1/messages endpoint exposed by a companion “bridge” process. The system decomposes into three principal strata: (i) a thin transport layer (the bridge), (ii) a bounded, budget-governed agent loop (run.js, ~1,370 lines), and (iii) a deep tool pipeline module that owns the complete lifecycle of tool execution, from permission evaluation through crash-recoverable effect journaling. We describe the loop's per-turn phase structure — instruction-delta detection, a multi-stage context-compaction ladder, prompt-cache budget placement, request construction, response accounting, and a two-phase (read-parallel, write-serial) tool execution model with a mid-turn abort checkpoint — and formalize the safety invariants the implementation maintains, notably effect pairing (intent/result journaling around every side effect), plan-mode write fabrication, and the failure-streak escalation protocol. We argue that the design's central contribution is the separation of loop-level policy (budgets, cycle detection, stop reasons) from effect-level mechanism (execution, sinks, recovery), joined at exactly one seam: the midTurnCheck callback.

1. Introduction

Agentic coding systems iterate a simple contract: a language model receives a conversation and a tool catalog, emits either prose or tool_use blocks, and the harness executes tools and feeds results back until the model produces a final textual answer. While the contract is simple, production-grade harnesses must additionally solve bounded execution (token, cost, wall-clock, and step budgets), crash recovery (side effects must be journaled before they occur), context management (histories exceed model context windows), and permissioned execution (writes and shell access require policy and, at times, human confirmation).

The Claude Local Bridge Runner addresses all four concerns in a single-process Node.js harness. Its domain language is codified in CONTEXT.md: the runner is the local coding-agent loop driven from bin/local-bridge-runner.js; the bridge is a transport-only VS Code extension exposing Claude Code credentials as a local HTTP endpoint; the agent loop is the per-turn cycle inside src/runner/run.js; and the tool pipeline is the deep module owning everything between “the model emitted tool_use blocks” and “completed, recorded tool results exist.” This paper documents these layers as implemented, with emphasis on the agent loop's control flow and its invariants.

2.3 Design stance

The architecture follows a “deep module” philosophy: run.js is a long but shallow orchestrator whose complexity is sequential, not combinatorial, while tool-pipeline.js hides a large behavioral surface (permissions, confirmation, plan-mode fabrication, parallelization, journaling, nine-step sink ordering) behind a two-method interface: toolDefinitions() and executeTurn(step, toolUses, { midTurnCheck }). A notable corollary documented in the pipeline header: the tool definitions shown to the model are derived from the same context object that gates execution, so the model can never see a tool the pipeline would refuse to run.

3. Run Lifecycle: Initialization

Before the loop begins, run(options) executes a strictly ordered bootstrap. Each stage can veto the run with a typed stop reason and exit code 1.

  1. Bootstrap & workspace trust. When kernel bootstrapping is enabled, boot validates the working directory, resolves its realpath, evaluates workspace trust, and loads instruction memory (project instruction docs plus a content hash). A blocked boot short-circuits with STOP_REASONS.CWD_INVALID.
  2. Fork-depth guard. assertForkAllowed(ctx.spawnDepth) prevents unbounded recursive agent spawning.
  3. CWD confinement. safety.validateCwd re-validates and pins ctx.cwdRealpath; all subsequent path confinement anchors here.
  4. Identity & persistence. A trace id (runId) is minted; the SessionStore (message checkpoints) and SessionLedger (effect journal) are opened if a session path resolves. The ledger immediately records session_started and surfaces any pending intents from a crashed prior run (“crash recovery” hint).
  5. Hooks. A HookDispatcher is constructed against .bridge-runner/ and session_start is dispatched.
  6. Budget tracker. createBudgetTracker merges explicit token budgets with budgets inherited from a parent agent via BRIDGE_RUNNER_BUDGET_INPUT_REMAINING / _OUTPUT_REMAINING environment variables — child agents therefore cannot exceed the parent's remaining allowance.
  7. Flight recorder. A JsonlTrace (levels normalized by normalizeTraceLevel; off disables) records boundary-annotated events; run_started captures flags, artifact paths, and an explicit visibility-notes partition of what only the runner, only the bridge, or only the upstream can observe.
  8. Archive collector. When enabled, a RunArchiveCollector snapshots prompt, flags, and artifact paths for end-of-run export.
  9. Tool pipeline construction. createToolPipeline({ ctx, runId, confirm, sinks, verbosity, failureLimit }) is built once per run; the sink bag (ledger, hooks, output, trace, transcript, humanLog, archive — each nullable) and the confirm port are fixed for the run's lifetime.
  10. Message hydration. Three mutually exclusive paths:
    • Session resume: load the store, hydrate runner state, run assertResumeAllowed (a degraded-health session requires explicit risk acknowledgment), copy prior messages, append the new user message.
    • Transcript resume: deprecated; refused with RESUME_FAILED.
    • Fresh: a single user message built from prompt and stdin, optionally prefixed by a dynamic-environment block when the context policy excludes dynamic data from the system prompt (a prompt-cache-friendliness measure). --new-session resets health, compaction generation, and failure counters.
  11. Prompt assembly. resolveSystemPrompt (progressive, policy-driven, override/append-capable) plus a separately cached repo-context block; tool definitions from the pipeline; maxSteps defaults applied; effort normalized.

4. The Agent Loop in Detail

The core is a bounded for (let step = 1; step <= steps; step++) loop. One iteration (“step” or “turn”) performs up to eleven phases. We enumerate them in execution order; phases 1–5 are pre-flight, 6–8 concern the model exchange, and 9–11 concern tool execution and continuation.

4.1 Phase 1 — Hook dispatch

hooks.dispatch('pre_model_request', { step, runId }) gives user hooks a synchronous observation point before any request work.

4.2 Phase 2 — Instruction-delta detection

Instruction files can change while the agent runs (e.g., the user edits CLAUDE.md mid-session). instructionDelta.detectChange(cwd) classifies the change:

  • small_diff: a compact delta block is injected as a user message — cheap, preserves the prompt cache for the (unchanged) system prompt.
  • large_rewrite: instruction memory is reloaded, its hash refreshed, and the entire system prompt is rebuilt via resolveSystemPrompt — correctness over cache economy.

Both paths are recorded in the transcript as typed instruction_delta entries.

4.3 Phase 3 — The compaction ladder

applyCompactionLadder(messages, system, policy) applies progressively more aggressive, staged reductions to the conversation when it exceeds policy thresholds (the compaction markers visible in compacted transcripts — e.g. [compaction:snip] for removed tool output and ghost-generation summaries — are its artifacts). Key properties:

  • It returns a possibly new messages array and a possibly modified system (systemForRequest); the loop rebinds both.
  • A monotonically increasing compaction generation is persisted to the session store, so resumed sessions know how degraded their history is; the generation feeds session-health scoring.
  • Applied stages are fanned out to the output event stream, the ledger (compaction_applied), and a stderr hint — compaction is never silent.

4.4 Phase 4 — Cache placement and advisory estimation

applyCacheControlBudget(systemForRequest, tools, messages, repoContextBlock) allocates Anthropic's limited cache_control breakpoints across the system prompt, tool definitions, the repo-context block, and stable message prefixes, returning { cachedSystem, cachedTools, cachedMessages }. Separately, estimateTokensAdvisory(messages) produces a local (non-authoritative) size estimate used only to emit a predictive warning when nearing maxContextTokens — the authoritative check happens post-response with real usage numbers (§4.7).

4.5 Phase 5 — Request construction

const requestBody = {
  model, max_tokens: maxTokens,
  system: cachedSystem, messages: cachedMessages, tools: cachedTools,
  ...(stream && outputFormat === 'text' ? { stream: true } : {}),
  ...(typeof temperature === 'number' ? { temperature } : {}),
  ...(effortLevel ? { output_config: { effort: effortLevel } } : {}),
};

Streaming is enabled only for plain-text output format (structured output formats need the whole response object). The built body is traced at the runner_to_bridge boundary with both a summary and a level-gated full payload capture.

4.6 Phase 6 — The model exchange and bridge-error policy

The request goes through modelClient.postStream (streaming) or modelClient.post, carrying per-step trace correlation headers and an optional caller token. On transport failure the loop implements a shared-streak retry: pipeline.recordExternalFailure() increments the same consecutive-failure streak used for tool failures, and the loop continues to the next step (consuming a step as the retry budget) while the streak is below MAX_CONSECUTIVE_FAILURES. Once at the limit — or on the final step — the run terminates with BRIDGE_ERROR. Unifying bridge and tool failures in one streak is deliberate: it bounds total consecutive dysfunction, whatever its source.

4.7 Phase 7 — Accounting and budget boundaries

After post_model_response hooks and sink recording, usage is accumulated:

totalUsage = addUsage(totalUsage, response.usage);
const budgetStopAfterModel = emitBudgetBoundary(step, totalUsage);

emitBudgetBoundary asks the budget tracker for a verdict, emits a budget event and trace record, syncs remaining-budget figures into ctx (so child agents spawned later inherit correct remainders), surfaces warnings, and returns a stop directive when a hard budget is exhausted. It is invoked at three boundaries per step: after the model response, inside the mid-turn check (before writes), and after the turn completes. A separate coarse guard stops the run outright if consumed context tokens exceed 2 × maxContextTokens (CONTEXT_BUDGET_EXCEEDED), a backstop against runaway cost when compaction cannot keep up.

4.8 Phase 8 — History append and persistence

The assistant message is appended to messages and the session is checkpointed via persistSession (which also folds the current failure streak into persisted runner state). Persistence occurs after every history mutation — assistant append and tool-result append alike — so a crash at any point resumes at a message-granular boundary, with the ledger's intent/result pairing (§6.1) covering the sub-message effect window.

4.9 Phase 9 — Extraction and the success exit

extractTextBlocks and extractToolUses partition the response content. Two policy gates follow:

  • Per-turn tool cap: if toolUses.length > maxToolCallsPerTurn, the run stops with MAX_TOOL_CALLS_PER_TURN before executing anything — a runaway-fanout guard.
  • Success: if toolUses.length === 0, the model has answered. The loop writes finals to transcript/human log, emits a result event, dispatches session_end, appends run_stopped to the ledger, builds a loop autopsy (a behavioral post-mortem over toolHistory: streaks, cycles, usage shape) written beside the session, optionally triggers session-extract, persists, archives, and returns through completeRun (§8.1).

4.10 Phase 10 — Delegation to the pipeline and the mid-turn check

The loop hands the turn to the pipeline with a single injected callback:

const turn = await pipeline.executeTurn(step, toolUses, {
  midTurnCheck: (readOutcomes) => {
    // record reads into toolHistory, then:
    //   detectSemanticCycles(toolHistory) → SEMANTIC_CYCLE_DETECTED
    //   wall-clock exceeded              → WALL_CLOCK_BUDGET_EXCEEDED
    //   estimateCostUsd > maxCostUsd     → COST_BUDGET_EXCEEDED
    //   emitBudgetBoundary(step, usage)  → token-budget stop
    return null; // or { stop, message }
  },
});

This is the architecture's key seam. The pipeline invokes the callback exactly once — after the read-only batch has executed and been recorded, but before any write effect. Loop-level state (tool history, budgets, clocks) never leaks into the pipeline; the pipeline exposes only the checkpoint moment. If the verdict stops the turn, executeTurn returns an aborted descriptor with the already-collected read results, and the loop finalizes with the given stop reason (semantic-cycle stops additionally produce an autopsy).

4.11 Phase 11 — Continuation

Tool results are appended as a single role: 'user' message whose content is the array of tool_result blocks; the session is persisted; the loop proceeds. If the for loop exhausts steps without a text-only response, the run ends with MAX_STEPS.

5. The Tool Pipeline

5.1 Two-phase execution model

executeTurn first validates shape (every tool use must have id and name; violation is one of the few conditions under which it rejects), then partitions tool uses by CATEGORIES[name]:

  • Read phase (parallel): all read-only tools execute as one batch via executeReadOnlyBatch. Effect ids are minted and tool_effect_intent records appended before the batch runs; results are paired afterward. Sibling failures annotate surviving results rather than aborting the batch.
  • Write phase (serial, model-emitted order): each write/shell tool runs one at a time, preserving the order the model emitted.

5.2 Opportunistic write parallelism (B3)

Under --accept-edits — and never in plan mode — consecutive writes whose canonicalized paths are pairwise disjoint may pre-execute in parallel. Disjointness (_arePathsDisjoint) requires that no path equals or prefix-contains another (with a separator boundary, catching parent/child conflicts); paths are canonicalized through safety.confinePath, whose realpath anchoring defuses symlink aliasing at the source. Groups execute via Promise.all(executeForce(...)), results are cached by tool_use_id, and the serial loop then consumes cached results in model-emitted order — so every sink observes a linear history indistinguishable from serial execution. A tool_use_group ledger record documents the parallel grant. The plan-mode exclusion is explicitly justified in the source: executeForce would bypass plan-mode fabrication and write for real.

5.3 Plan-mode fabrication

In plan mode, any tool result flagged needsConfirmation is replaced in exactly one place by { ok: true, text: 'Plan mode: would ' + proposedAction }. The model receives a plausible success narrative; the filesystem receives nothing.

5.4 Failure streak

The streak counts consecutive failed write/shell executions (read failures historically do not escalate) plus bridge transport errors (recordExternalFailure). At failureLimit, an escalation notice is appended to the failing result's text before any sink records it — guaranteeing the transcript, human log, ledger, and the model itself all see the identical escalated text. Any successful write resets the streak. The loop reads pipeline.failureStreak back for session persistence, and resumed sessions re-seed it via initialFailureStreak.

5.5 Sink fan-out ordering

Per tool use, sinks are notified in a fixed nine-stage order:

ledger intent → pre_tool hook → tool_use event → tool_requested trace
      → tool_call transcript → [execute] → tool_result transcript
      → human log → tool_finished trace → archive → ledger result
      → post_tool hook → tool_result event

The ledger is the sole critical sink: an append failure aborts the turn, because an effect that cannot be journaled would break crash recovery. Every other sink is best-effort — failures are reported to stderr and never alter tool results.

6. Safety Invariants

Invariant 1 (Effect pairing). Every tool use appends a ledger tool_effect_intent with a fresh effectId before execution, and exactly one tool_effect_result with the same effectId after — on success, throw, permission deny, user denial, and plan-mode fabrication alike. An intent without a result is therefore a machine-detectable crash signature, surfaced at next startup as “pending effects.”
Invariant 2 (Plan-mode write safety). In plan mode no write tool executes; the parallel pre-execution path is disabled outright, and confirmations are fabricated in a single code path.
Invariant 3 (Observation linearity). Regardless of physical execution order (parallel read batch, disjoint-write pre-execution), all sinks record events in the model's emitted order. Replay consumers may assume a linear history.
Invariant 4 (Catalog/execution coherence). toolDefinitions() derives from the same ctx that gates execution: the model is never shown a tool the pipeline would categorically refuse.
Invariant 5 (Mid-turn write barrier). Loop-level stops (semantic cycle, wall clock, cost, token budgets) are evaluated after reads are recorded and before any write executes; a stopped turn performs no side effects beyond its reads.
Invariant 6 (Budget inheritance). Child agents receive the parent's remaining token budget via environment, and the parent re-syncs remainders after every boundary; the family's aggregate consumption is bounded by the root budget.

8. Observability and Persistence

8.1 Finalization (completeRun)

A single closure owns end-of-run bookkeeping for every terminal path: the usage/cost summary (stderr-only, keeping stdout clean and pipeable), transcript and human-log usage records, a session health verdict (from stop reason, autopsy, compaction generation, and failure streak — possibly recommending a fresh session), synchronous session flush, ledger close, and archive export. Centralizing this in one site is an explicit design note in the source: one owner, all paths covered.

8.3 Boundary-aware tracing

Trace records are tagged with explicit boundaries (runner_to_bridge, bridge_to_runner), and run_started documents a visibility partition: the runner alone sees cwd validation, permission decisions, and local effects; the bridge alone sees credential sourcing and header injection; only the upstream-bound payload contains the transformed Messages body. Per-step correlation headers (bridgeTraceHeaders) let the two layers' traces be joined offline by runId and turn.

9. Discussion

Three design decisions merit emphasis. First, the single-seam decomposition: the only channel through which loop policy influences in-flight tool execution is midTurnCheck, invoked at a precisely specified moment. This keeps the pipeline testable in isolation (scripted confirm ports, deny-all adapters for coordinator workers) while giving the loop a genuine write barrier for its budget and cycle guards. Second, the asymmetric sink contract — one critical journal, many best-effort mirrors — buys rich observability without letting a logging failure corrupt agent behavior, while effect pairing turns the journal into a crash-recovery oracle. Third, economics as a first-class loop concern: prompt-cache breakpoint budgeting, a staged compaction ladder with a persisted generation counter, predictive-versus-authoritative token checks, inheritable child budgets, and dollar-denominated cost stops are all evaluated inside the per-step control flow rather than bolted on as middleware.

The principal cost of the design is the length of run.js: eleven phases and roughly a dozen exit paths in one function, each duplicating result-shape construction. The codebase acknowledges the trade — sequential orchestration reads top-to-bottom, and the deep modules it calls into absorb the combinatorial complexity. Future simplification would most plausibly extract a terminal-result builder to collapse the duplicated exit blocks without disturbing the phase ordering that the invariants depend on.
