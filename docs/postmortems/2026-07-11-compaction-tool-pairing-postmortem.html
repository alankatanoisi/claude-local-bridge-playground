Compaction split a tool call from its result

A max-settings Fable 5 build run exposed an unsafe history boundary in the Claude runner. The work survived; the request did not.

Executive verdict

The max settings exposed a real runner defect; they did not make an otherwise-correct runner entitled to send malformed history. At step 52, summarizeOldTurns() kept the last 12 raw messages as if they were always six complete turns. A standalone instruction-update message shifted that cutoff, leaving a tool_result whose matching tool_use had been summarized away.

Anthropic correctly rejected the request with HTTP 400. A tiny local harness reproduces the same failure without a model, bridge, network, large prompt, or long run.

Root cause in the code

The compactor says it preserves recent “turns,” but it actually subtracts twice that number from the raw message count: 

const cutoff = Math.max(0, messages.length - preserveRecent * 2);
const head = messages.slice(0, cutoff);
const tail = messages.slice(cutoff);
return { messages: [summaryMsg, ...tail], ... };

This assumes every logical turn is exactly an assistant message followed by a user message. The instruction-delta feature can add a standalone user message before compaction. Once the alignment shifts, an arbitrary raw-message cutoff can land on a user tool_result.

Missing defense: no request-boundary validator checks that every retained Anthropic tool_result is immediately preceded by a matching assistant tool_use.

What the max settings actually did

max_steps=1000

The main stress enabler. The default is 16; this run needed 52 turns to reach full summarization.

max_tokens=128000

A per-request output ceiling, not the compactor threshold. It created headroom but did not define valid history.

Important UI/documentation gap: --max-context-tokens does not configure the compaction ladder. Current code uses it for separate advisory/cumulative-stop logic. The compactor retains its own 80,000/160,000 thresholds unless a preset overrides them.

Conclusion: this was a successful stress test in the engineering sense—it exposed a hidden invariant violation.

Test gap and secondary defects

False-green pairing test

The existing test is named “summarizeOldTurns preserves recent turns and pairs,” and its own fixture produces an orphaned older result. It passes because it checks only that one newest result exists, not that every result has the correct immediately preceding call.

Non-idempotent clipping

The clip function keeps 12,000 characters and then adds a footer, leaving content above the same limit. A second pass clips it again. This explains 39 clip-only records; they were not 39 full summaries.

Unproductive retry

A deterministic invalid-request 400 is retried twice without changing history. Local validation and non-retriable error classification would save time and requests.

Two repositories, two native fixes

Claude playground — owns this incident

  • Fix semantic grouping in the Anthropic-message compactor.
  • Add instruction-delta integration coverage.
  • Validate tool pairing before each request.
  • Keep bridge/auth/proxy internals untouched.

Codex fork — learns from it

  • Do not reactivate or blindly cherry-pick the Anthropic compactor.
  • Keep function_call plus function_call_output atomic.
  • Preserve native reasoning-item order and adjacency.
  • Log deliberate cross-repo ports in PORTING.md.

This is behavioral convergence, not schema convergence: both runners need the same logical guarantee, implemented in their provider-native history formats.

Codex roadmap consequence

The current roadmap schedules live compaction in Phase 4 but basic compaction goldens in Phase 5. This incident shows that order is unsafe for a load-bearing history transform.

  1. Keep Stage 7 focused on pricing, documentation, and one short live read-only run.
  2. Before the Phase 4 compaction rung, implement a native Responses compactor—or explicitly block that rung.
  3. Move basic native invariants before live compaction: call/output atomicity, reasoning adjacency, standalone instruction messages, multi-call batches, resume/replay, and post-compaction request validation.
  4. Use Phase 5 for adversarial, fuzz, and broader hardening—not the first pairing test.
