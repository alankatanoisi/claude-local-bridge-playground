Local Bridge Runner

A local coding agent that uses your existing claude-local-bridge as its model transport. The playground is OAuth-only: no Anthropic Console API key is used upstream.

CLI Runner Agent OS (playground)

What it does

The runner sends your prompt to the local bridge (usually on localhost:11437), which forwards it upstream using a Claude Code OAuth Bearer token resolved on your Mac. When the model asks to read a file, list a directory, or search text, the runner executes the request safely and sends the result back.

Key benefit You can run agent-style queries from the terminal without setting up a separate Anthropic API key. The bridge intentionally ignores API-key fallback so the OAuth policy experiment stays clean.

In the playground checkout, the runner is layered like a small agent OS: an optional coordinator runs phased work (research → synthesize → execute → verify), the kernel is the core tool loop, and a session store keeps canonical *.state.json files for reliable resume.

Before you start

Make sure the bridge is running. In VS Code, start the extension (or press F5 in development mode). Then verify: 

curl -s http://localhost:11437/v1/models | python3 -m json.tool

Look for a JSON model list. If you need diagnostics, /v1/debug now requires the local debug token printed in the Claude Local Bridge Output log: 

curl -s http://localhost:11437/v1/debug \
  -H "x-claude-local-bridge-debug-token: PASTE_TOKEN_HERE" \
  | python3 -m json.tool

Look for "credentialPolicy": "oauth-only" and "authenticated": true.

Caller auth is optional. If you intentionally enable it in VS Code settings, also set the local caller token in Terminal before runner commands: 

export BRIDGE_CALLER_TOKEN=local-dev-token

That token authorizes your local terminal to call your local bridge. It is not your Anthropic credential.

Quick start in 3 steps

1

Ask a simple question (no tools needed)

cd "/Users/alanman/Developer/claude-local-bridge-playground"
node bin/local-bridge-runner.js "What is 2 + 2?"

The model should answer immediately with no tool calls.

2

Ask about your project (triggers inspection tools)

node bin/local-bridge-runner.js \
  --verbose \
  "List the main files in this repo and explain what each does"

You will see step-by-step progress in stderr as the model calls list_files and read_file.

3

Check the transcript

ls ~/.bridge-runner/logs/

Every run writes a timestamped JSONL file so you can review what the model did.

In the playground, also try --session-id my-run — session state is saved separately at ~/.bridge-runner/sessions/my-run.state.json.

Testing Other Local Folders

The safest pattern is to run the runner from this repository, then point it at another project with --cwd. Think of it as two folders:

Folder What it means
Runner repo folder The folder that contains bin/local-bridge-runner.js.
Target project folder The folder you want the model's tools to inspect or edit. This is passed as --cwd.

Read-only smoke test against any project

cd "/Users/alanman/Developer/claude-local-bridge-playground"
node bin/local-bridge-runner.js \
  --cwd "/Users/alanman/path/to/another/project" \
  --max-steps 8 \
  --verbose \
  "List the top-level files, summarize what this project does, then stop. Do not edit files."

If it worked, you should see verbose lines like [runner] step 1: tool_call list_files, followed by a final summary. If the model loops, lower --max-steps and make the prompt more explicit.

Attach files from that target project

cd "/Users/alanman/Developer/claude-local-bridge-playground"
node bin/local-bridge-runner.js \
  --cwd "/Users/alanman/path/to/another/project" \
  --include-file README.md \
  --include-file package.json \
  "Review the attached files and explain the project setup. Do not edit files."

Use stdin from that target project

cd "/Users/alanman/Developer/claude-local-bridge-playground"
cat "/Users/alanman/path/to/another/project/README.md" | node bin/local-bridge-runner.js \
  --cwd "/Users/alanman/path/to/another/project" \
  "Summarize this README and suggest one next thing to inspect."
Avoid the old prototype folder If you run from claude-local-bridge-runner-test, you may be using stale code. Use the clean worktree path above, or replace it with the real folder that contains the current runner.

Available tools

The runner exposes read tools by default and can also expose guarded edit and shell tools when you enable them.

list_files

List files and directories under a relative path. Skips .git, node_modules, dist, build, and coverage by default.

read_file

Read a file by relative path. Respects byte and line limits. Blocks secret-looking files (.env, *.pem, credentials*.json, etc.).

search_text

Search for text inside the project. Prefers ripgrep, falls back to grep, then to a pure Node walk. Case-insensitive.

git_status

Run git status --short. Read-only. Never adds, commits, resets, or pushes.

edit_file / write_file / apply_patch

Mutating tools for exact replacements, full writes, and patch-based changes. Writes are atomic, edits can use a SHA-256 guard, and they ask for confirmation unless you pass --accept-edits.

bash

Optional shell tool with timeouts and output limits. It is hidden from the model unless you pass --allow-shell.

undo

Restore file backups created by write tools. This is the recovery path if an edit goes wrong.

undo_edit

Restore the backup from a specific current-run edit or write by tool_use_id, or undo the most recent change for a path.

Flight recorder traces

Use --trace-level summary when you want evidence about a run without logging prompt bodies. The runner writes a JSONL trace under ~/.bridge-runner/traces/ and asks the bridge to write a correlated trace under ~/.claude-local-bridge/traces/.

Summary traces record turn counts, tool decisions, request sizes, visible tool names, response status metadata, and model usage or cache counters returned to the runner. redacted adds scrubbed request, response, tool-input, and tool-result payloads. full keeps the widest local payload evidence while still redacting authorization and key-looking fields.

These traces show what this local runner and bridge can observe. They do not expose Anthropic's internal detection logic or server-side telemetry. Treat redacted and full traces like source-code logs.

The runner adds correlation headers automatically. A direct Anthropic /v1/messages bridge client can instead set the VS Code setting claudeLocalBridge.traceLevel or send x-local-bridge-trace-level: summary with an authenticated local request.

Plan mode

Pass --plan to make the runner describe what it would do instead of actually doing it. In plan mode, all write and shell tools return dry-run messages like "Plan mode: would Write src/app.js". Read tools still execute normally so the model can inspect the project. 

node bin/local-bridge-runner.js \
  --plan \
  --max-steps 6 \
  "Describe what changes you would make to fix the failing test"

Plan mode is useful for reviewing the model's intentions before committing to changes. Combine it with --verbose to see each step in stderr.

Network control

Pass --no-network to add a best-effort HTTP and HTTPS proxy guard to shell commands. This sets http_proxy and https_proxy to 127.0.0.1:1 in the tool's environment, which prevents most curl, wget, and npm calls from reaching the internet.

Not a sandbox A shell command can unset proxy variables or use a protocol those variables do not control. Use --no-network as defense in depth, not as a promise that bash has no network. See the threat model for details.

Prompt caching and parallel execution

The runner sends cache_control: { type: "ephemeral" } on the system prompt each turn. If you are using the Anthropic API with prompt caching enabled, this reduces repeated instruction tokens on subsequent turns while staying under Anthropic's cache-control block limit.

Read-only tools (list_files, read_file, search_text, git_status) are batched before write and shell tools. This means the model's read requests execute faster without waiting for confirmation prompts.

Resuming sessions

Use --resume <path> to continue from a specific transcript, or --continue to automatically pick up the latest transcript from ~/.bridge-runner/logs/. 

node bin/local-bridge-runner.js --continue "Now add error handling to the function we just wrote"

In the playground, prefer --session-id for reliable resume. The JSONL transcript is an audit log; the *.state.json file under ~/.bridge-runner/sessions/ is the canonical conversation state when a session id is set. 

node bin/local-bridge-runner.js \
  --session-id my-project-1 \
  --resume ~/.bridge-runner/logs/2025-05-24T12-00-00.jsonl \
  "What did we decide in the last run?"

Context compaction (playground)

Long runs can exceed context limits. The playground compactor applies a ladder: clip (trim old tool output), snip (collapse middle turns), then ghost (summary placeholder). Set --max-context-tokens to get warnings before hard stops.

Custom system prompts

Pass --system-prompt "Your custom instructions" to replace the built-in system prompt entirely. This is useful for specialized workflows: 

node bin/local-bridge-runner.js \
  --system-prompt "You are a code reviewer. Only read files and provide feedback. Do not edit files." \
  "Review src/runner/run.js for potential bugs"

Safety rules

Mutations are gated Edit tools ask for confirmation by default. Shell is disabled by default. Unsafe paths and secret-looking files are blocked even when the mutation tools are enabled. For the full threat model, see threat-model.md.
  • Blocked paths: .env*, *.pem, *.key, *.p8, *.p12, credentials*.json, service-account JSON, token files, and anything that looks like a secret.
  • Blocked escapes: absolute paths and ../ outside the working directory are rejected.
  • Transcripts live in ~/.bridge-runner/logs/ (outside the repo) and redact sensitive headers.
  • Human-readable logs also redact secret-looking values before they are written to disk.
  • Write tools save backups in .bridge-runner/backups/ so the undo tool can restore them.

Troubleshooting

“Bridge returned HTTP 404” or connection refused

The bridge is not running or is on a different port. Check:

curl -s http://localhost:11437/v1/models

If that fails, start the bridge first (in VS Code: Claude Local Bridge: Start Server).

“No credentials found” in bridge debug

Open Claude Code in your terminal once. The bridge intercepts the live OAuth token from Claude Code's outgoing HTTPS requests.

Model loops forever

Use --max-steps to cap the loop. The default is 16. If the model keeps calling tools without reaching a final answer, try making your prompt more specific: 

"List the files and then stop. Do not call any more tools after listing."

Approval was denied automatically

Write and shell approvals need an interactive Terminal prompt. If another program launches the runner without a usable terminal, the runner denies the action instead of guessing. Re-run it directly in Terminal, narrow the task to read-only tools, or intentionally opt in with flags such as --accept-edits.

Transcript files filling up

They are small JSONL files in ~/.bridge-runner/logs/. You can delete old ones safely:

rm ~/.bridge-runner/logs/*.jsonl
