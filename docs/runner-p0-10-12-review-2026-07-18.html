P0-10 — Cached permissions & recovery targets survive a root change CONFIRMED

ClaimEvidence (current code)
Decision-cache key omits root identitysrc/runner/permissions.js _decisionKey() (~lines 130–139): key = tool name + flag bits + normalized args. No cwd/worktree/root token anywhere in the key.
Worktree transitions don't clear decisionssrc/runner/worktree-utils.js activateSlot() / deactivateToRepoRoot() mutate the same ctx object (ctx.cwd = entry.path, ctx.cwdRealpath = ...). The per-ctx WeakMap cache therefore survives intact. invalidateDecisionCache is only ever called from tool-registry.js on specific write paths — never on root transitions.
undo_edit validates one path, writes anothersrc/runner/tools/undo-edit.js (~lines 68–79): confinePath(entry.path, ctx) validates against the current root, then the restore writes entry.absolute_path — the absolute path recorded under the old root. Validation target and write target diverge.
Blindspots beyond the assessment:
  • safety.js keeps a per-ctx cachedRealpathSync cache that is also never cleared on root transitions (lower risk — realpaths are root-independent — but stale after renames/moves inside a worktree swap).
  • test/runner/permission-decision-cache.test.js exercises per-ctx isolation and flag changes but has no worktree/root-change case at all — the exact regression this P0 covers is untested.

Recommended shape: add a monotonic rootEpoch on ctx; include it in _decisionKey; bump it in activateSlot/deactivateToRepoRoot (O(1), no cache walk). In undo-edit.js, make the confined path the single source of truth for the write target and treat absolute_path as advisory metadata only.

  • Reclassification: the manifest-despite---no-session-persistence behavior is now intentional and documented in run.js:333–338 (syncRunManifest powers undo last-run). Keep the behavior; give the manifest 0600/0700 modes and keep its content minimal (paths + hashes, no message text).

  • Test gap: session-store/ledger tests exist (session-ledger.test.js, session-store-debounce.test.js, etc.) but none assert file modes.

3. Global / performance observations

  • Test suite wall time: full npm test exceeds the 30s harness cap on every run; test/runner/ alone holds 89 files. All observed suites pass. Recommend: a tagged smoke subset (<10s) for inner-loop verification + sharded/parallel full run for handoff checks. This is a small, high-leverage P1.
  • Cache-fix pattern: prefer epoch tokens over cache clears throughout (decision cache, realpath cache) — O(1), no lock/ordering concerns, and self-documenting in keys.
  • Redaction throughput: centralizing at the emit chokepoint means every output byte flows through scrubber regexes once — acceptable; avoid double-scrubbing (results scrubbed in tool-registry, then again at emit). Consider a "already-scrubbed" marker or scrub only at the boundary and drop the inner call after migration.
  • Ordering rationale (P0-10 → P0-12 → P0-11): P0-10 is small and self-contained (2 files + tests). P0-12 lands a privateWrite helper that P0-11's persistence sinks then reuse. P0-11 is the widest sweep and goes last with the most infrastructure already in place.
