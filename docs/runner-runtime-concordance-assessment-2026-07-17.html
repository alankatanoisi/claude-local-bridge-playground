Posterity: decisions and why

  • Batch contract: tool results need not complete FIFO, but every assistant tool batch must be followed immediately by one complete user result batch with exactly matching IDs.
  • Foreground flexibility: after three consecutive fully failed tool batches, the runner asks whether to continue, add guidance, or stop. A multi-tool batch counts once, any success resets the streak, and a user's denied permission is not a tool malfunction.
  • Less is more: Alan relies on granular flags and had not used the profile systems. Archival retirement removes conflicting authority layers now while leaving historical material available for a future, separately designed narrowing-only subagent-template concept.
  • P0-04 interpretation: realpath confinement controls where a tool may look; the deny matrix controls sensitive file classes it may never read, including inside the authorized root. Model judgment alone cannot expand either boundary.
  • P0-06 quarantine then repair: refuse the unsafe shell surface immediately; later replace with pure-JS apply (no shell), full hunk validation, atomic write, and rollback — still hidden unless --tools apply_patch.
  • P0-09 honesty over pretend sandbox: document shell as host-account authority; keep regex scanning and --no-network as defense-in-depth; portable OS isolation is a separate project.
  • P0-12 before P0-11: private-fs lands first so redaction persistence sinks write privately by construction.
  • P0-11 display vs execute: scrub sink copies of tool inputs; keep raw inputs for write_file so project content with token-like strings stays byte-exact on disk.

Approval and final acceptance checklist

After the implementation packages—not during this review gate—the repository should satisfy all of the following:

  • Message integrity: semantic compaction fixtures cover odd/even histories, instruction deltas, multi-tool batches, repeat application, resume/replay, clipping idempotence, and the exact postmortem regression.
  • Authority: every CLI permission mode × agent × tool profile × capability group × parent/child combination stays below the explicit ceiling.
  • Security: denied files remain invisible through every search backend and symlink; hidden/aliased tools fail; root changes invalidate caches; recovery cannot cross roots; patch filenames cannot execute; secret fixtures never reach any sink.
  • Lifecycle: every stop/abort path produces a complete, correlated private bundle and exactly one final event.
  • Compatibility: fixtures cover version/help changes, stale fingerprints, header classes, supported effort/thinking/sampling combinations, typed 400/429/5xx behavior, and offline/stale metadata.
  • Built-ins: every shipped agent/profile/template/skill loads from a clean clone and requests only intended tools.
  • Performance: representative task results are compared before changing effort defaults.
  • Live acceptance: safe-core inspection, evidence-capable plan mode, multi-turn tools, current-model adaptive thinking, deterministic 400 fail-fast, transient retry, and runner/bridge trace correlation all pass.
  • Final checks: full tests, runner tests, lint, documentation checks, formatting, prompt/agent/profile validation, clean Git status, and packaged browser verification pass.

Limitations and uncertainty

  • The Sonnet 4.6 HTTP 429 was live and repeatable, but attributing it to the blanket 1M-context beta header is an inference, not a proven causal result. The approved compatibility probe must isolate headers and remain token/cost capped.
  • Official public model documentation establishes catalog and parameter rules, not this account’s entitlements. Account-specific availability must never be inferred from a public catalog alone.
  • Static code confirmation was used instead of executing destructive or credential-seeking exploits for apply_patch, worktree cleanup, shell escape, and recovery-path risks.
  • The schema token numbers use the report plan’s simple four-bytes-per-token estimate. Actual tokenizer and prompt-cache effects vary; bytes are the stable regression metric.
  • Browser verification covers the packaged reader’s desktop and narrow Chromium smoke, not Safari/Firefox/touch/pixel-parity certification.

These limitations do not change the containment ordering because the P0 recommendations either close directly reproduced behavior or remove code-confirmed high-impact execution paths.
