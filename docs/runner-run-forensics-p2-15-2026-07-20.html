The one-paragraph story. All four runs faced the same intermittent upstream failure (empty-body 500 internal_error from Anthropic, in bursts). Run 1 (Fable) died because three failures landed consecutively and exhausted the retry budget of 2 — after 6.6 minutes of pure reading, zero writes. Run 2 (Opus) sailed through a clean window but died the moment it tried to write code, because the default 2,000-token output cap truncated its first big file write. Run 3 (Opus, cap raised) absorbed five 500s with retries, wrote the actual code in steps 24–26, and was then stopped by the --max-cost-usd 10 circuit breaker while tidying up. Permissions never once intervened: every one of the 103 tool calls was auto-allowed by the accept-edits mode policy.

1 · How to read a run (the vocabulary)

The runner is a loop. Each cycle is a step:

your prompt → model request → model replies (text + tool calls) → runner executes tools locally → results go back → next step

step
One model request. A step either succeeds (the model replied) or errors (the bridge/model call failed). Retries after an error consume new step numbers — that is why error steps and tool steps interleave below.
tool call
The model asking the runner to do something locally (read_file, search_text, edit_file…). Several can happen in one step.
permission decision
Before executing any tool, the runner’s policy engine records: category (read-only / write), the active mode, which rule decided, and the decision (allow / ask / deny). Every decision is archived per call.
compaction (“clip”)
When the conversation grows too big, the runner compresses older messages. Recorded in the ledger with an estimated token count.
wait time (in the charts)
Seconds from “previous step’s tools finished” to “this step’s tools started” — i.e. model thinking + network + any failed attempts in between. Local tool execution itself was never slower than 0.67 s all night, so waiting on the model is essentially all of the elapsed time.

3 · Run 1 (Fable 5): six minutes of reading, then the storm wins

Fable spent all 27 steps exploring the repo — 34 tool calls, all read-only (read_file ×18, search_text ×12, list_files ×4), zero edits. From step 14 the upstream started flapping. Single failures were absorbed (retry, next step succeeds), but the run only survives while failures come one or two at a time: the retry budget is 2 consecutive. At steps 25→26→27 came a 500, then two 120-second timeouts back-to-back — budget spent, run over. The two timeouts alone are 4 of the 6.6 minutes.

4 · Run 3 (Opus 4.8): the storm again — but this time the loop wins

Same task, same evening, five more 500s (two of them before the very first reply). Every burst stayed ≤ 2 deep, so retries absorbed all of them. The shape of the run is classic agent behavior: fast reads for ~3.5 minutes, then three big pauses — 65 s and 2 × 30 s — where xhigh-effort Opus composed the actual code, delivered as three edit_file calls (steps 24–26, orange). The cost breaker then ended the run at step 30 while it was updating its task checklist. The work survived: those three edits are what passed npm run check:docs and shipped as commit fc60801.

5 · The two short runs — each one a lesson

opus-run-2 · death by output cap model_max_tokens

Seven steps of brisk reading (median wait 3.4 s — Opus explores ~2× faster than Fable did on the same repo), then at step 8 the model began emitting its first big file write. The runner’s default is --max-tokens 2000 per response; a 175-line script doesn’t fit. The response was cut mid-tool-call and the runner — correctly — refuses to act on a truncated write rather than risk applying half a file. Cost of the lesson: $2.91.

Rule of thumb going forward: any run that is expected to create files needs both --capabilities edits (to have write tools at all) and a raised --max-tokens (so the write fits in one response).

opus-run-1 · death by operator kill — and an observability gap

This run was launched without --capabilities edits (write tools not offered), spotted within seconds via the runner’s own preflight tip, and killed. It did 15 read-only tool calls in ~27 s first. Because the process was stopped with a signal, it never reached its shutdown path — so it wrote no archive folder, no catalog row, no autopsy, and no usage/cost record. Only the ledger and session state survived (they are appended live). Its API spend happened but is invisible to every local accounting surface. See §7.

6 · Permissions: what fired, and what never got tested

Every archived tool call carries a full permission verdict. Across all three archived runs the record is remarkably uniform:

DecisionRuleCategoryModeSeverityCount
allowmode_policyread-onlyacceptEditsbypassable_ask85
allowmode_policywriteacceptEditsbypassable_ask3

Reading that row: the category is what the tool inherently is; severity bypassable_ask means “this would normally ask a human, but the current mode may bypass it”; and mode_policy means the blanket mode made the decision — no guard, no deny-matrix entry, no per-tool rule was ever consulted. Explanation string on every one: “Allowed by current permission mode.”

What this means for your evaluation: tonight exercised exactly one cell of the permission matrix (acceptEdits ⇒ allow-everything-offered). Zero confirmations, zero denials, zero guard hits, zero loop warnings. The deny matrix (.env, keys, path escapes), the confirmation flow, and plan mode are all still untested by real traffic. The experiments in §8 are designed to light those cells up one at a time.

Also true and worth knowing: with --accept-edits, nothing in the system prompt tells the model its writes are unreviewed — the guardrail is purely runner-side. The forked prompt-audit of these runs suggested a short “harness contract” in the prompt (output cap, step/cost budgets, who reviews writes); see §8, experiment E.

The dominant line item is cache reads: every step re-serves the whole conversation-so-far from cache (cheap per token, but the volume compounds — 30 steps × a growing context ≈ 1.35M tokens in run 3). Gap 1 — archive timestamps lie about timing. Every turn file in an archived run carries the flush-time, not the event time (all of fable’s 62 turns say 01:46:20). Timing truth lives only in the ledger. Fine once you know; a trap if you don’t.

Gap 2 — killed runs are financially invisible. Archive, catalog, autopsy and the usage line are all end-of-run writes; a SIGKILL leaves only ledger + state, and no cost anywhere. If you evaluate spend from the catalog alone you will undercount.

Gap 3 — no x-request-id on errors. The 500s were empty-bodied; the one identifier Anthropic could act on (the request id header) is captured by the trace layer, which was off, and the runner’s error line doesn’t include it. Turning on --trace-level summary during flaky periods costs little and makes failures reportable.
