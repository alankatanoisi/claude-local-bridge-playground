5 macOS Apple Shortcuts for the Bridge Runner

The local-bridge-runner.js CLI is a self-contained coding-agent loop: it talks to Claude through the local bridge on localhost:11437, and it exposes read, write, recovery, session, worktree, and (opt-in) shell tools against any folder you point it at with --cwd. That makes it a perfect backend for Apple Shortcuts Run Shell Script actions — following the exact pattern in your example screenshot (Receive input → Run Shell Script → Stop and output).

“Explain This Folder” — Finder Quick Action

Right-click any project folder in Finder → get an instant plain-English summary of what the codebase does.

Trigger: Finder Quick Action (folders) Read-only Uses: --cwd, explore template, repo map

Shortcut layout

  1. Receive Files/Folders from Quick Actions in Finder.
  2. Run Shell Script (zsh, Pass Input: as arguments) with the script below — $1 is the selected folder.
  3. Show Result (or Quick Look) with the Shell Script Result.
cd "/Users/alanman/Developer/claude-local-bridge-playground"
node bin/local-bridge-runner.js \
  --cwd "$1" \
  --prompt-template explore \
  --include-repo-map --include-repo-context \
  --dont-ask --no-session-persistence \
  "Summarize what this project does, its main entry points, and how to run it. Read-only; do not edit files."
Why it fits the runner: the built-in explore prompt template plus --include-repo-map gives the model structured context for unfamiliar repos, while the default read-only tool posture means a stray folder selection can never be modified. --dont-ask makes interactive confirmations fail closed instead of hanging the Shortcut.

“Review My Diff” — Pre-commit Code Review from the Menu Bar

One click before you commit: a findings-first code review of the current working tree using the shipped code-reviewer agent.

Trigger: Menu Bar / Hotkey Read-only (plan mode) Uses: --agent code-reviewer, review template, git_status

Shortcut layout

  1. Choose from List — a static list of your frequently-used repo paths (or an Ask for Input with a default).
  2. Run Shell Script with Chosen Item as input, passed as arguments.
  3. Show Result, and optionally Copy to Clipboard so findings can be pasted into the commit message or PR.
cd "/Users/alanman/Developer/claude-local-bridge-playground"
node bin/local-bridge-runner.js \
  --cwd "$1" \
  --agent code-reviewer \
  --prompt-template review \
  --prompt-arg focus="uncommitted changes shown by git_status" \
  --plan --dont-ask \
  "Run git_status, read every modified file, and review the pending changes for correctness, safety, and missing tests. Lead with concrete findings."
Why it fits the runner: the playground ships a curated code-reviewer file agent under .bridge-runner/agents/, and the review template is parameterized with a {{focus}} placeholder filled via --prompt-arg. --plan guarantees the review only describes changes — it never makes them.

“Explain This Screenshot” — Share Sheet Multimodal Analysis

Share any screenshot, diagram, or PDF from Preview/Photos/Finder and have the runner read it with its multimodal read_file tool.

Trigger: Share Sheet (Images & PDFs) Read-only Uses: multimodal read_file (7MB img / 10MB PDF caps)

Shortcut layout

  1. Receive Images and PDFs from the Share Sheet.
  2. Save File — write Shortcut Input to a scratch folder, e.g. ~/Documents/runner-inbox/ (the runner can only read inside its --cwd).
  3. Run Shell Script with the saved file path as argument.
  4. Show Result / Quick Look.
cd "/Users/alanman/Developer/claude-local-bridge-playground"
FILE="$1"
node bin/local-bridge-runner.js \
  --cwd "$HOME/Documents/runner-inbox" \
  --bare --dont-ask --no-session-persistence \
  "Use read_file on '$(basename "$FILE")' and explain what it shows. If it contains code or an error message, transcribe it and suggest a fix. If it is a UI, describe the flow."
Why it fits the runner: read_file natively attaches PNG/JPG/GIF/WebP/PDF bytes as Anthropic multimodal content blocks (exactly how your example screenshot was read for this report), so the Shortcut becomes a general-purpose “explain this image” tool — error dialogs, architecture diagrams, PDFs of docs, even other Shortcuts screenshots. --bare keeps the prompt minimal and fast.

“Siri, Fix It in a Worktree” — Dictated Task, Isolated Branch

Dictate a coding task to Siri; the runner performs the edits inside an isolated git worktree branch so your main checkout is never touched.

Trigger: Siri / “Hey Siri, fix it” Writes — but only in a throwaway worktree branch Uses: enter_worktree, --accept-edits, run manifests

Shortcut layout

  1. Dictate Text (or Ask for Input) — the task description.
  2. Run Shell Script with the dictated text as $1.
  3. Show Notification with the result summary, including the worktree branch name to review with git diff later.
cd "/Users/alanman/Developer/claude-local-bridge-playground"
node bin/local-bridge-runner.js \
  --cwd "/Users/alanman/Developer/my-active-project" \
  --accept-edits --dont-ask --max-steps 24 \
  "First call enter_worktree to create an isolated worktree branch. Then, inside it: $1. When done, summarize the changes and report the worktree branch name. Do NOT call exit_worktree with cleanup — preserve the branch for my review."

Rollback safety net if a run goes sideways: node bin/local-bridge-undo.js last-run --cwd <project> --dry-run

Why it fits the runner: this leans on the runner's headline safety features — enter_worktree puts all edits on a fresh branch under ~/.bridge-runner/worktrees/ (default cleanup=false preserves work for review), every edit is recorded in a per-run manifest.json, and local-bridge-undo can revert the entire run in one command. Hands-free coding with a real undo story.

“Morning Repo Health Check” — Scheduled Automation + Notification

Every weekday at 9 AM, spawn a verify-style pass over your active repo: dirty files, TODO drift, failing-looking tests, and doc mismatches — delivered as a notification digest.

Trigger: Shortcuts Automation (time of day) Read-only Uses: spawn_agent (verify), git_status, --trace-level summary

Shortcut layout

  1. Automation: Time of Day → 9:00 AM, weekdays, “Run Immediately”.
  2. Run Shell Script (no input needed).
  3. Show Notification with the Shell Script Result; optionally Append to Note “Repo Health Log” for a running history.
cd "/Users/alanman/Developer/claude-local-bridge-playground"
node bin/local-bridge-runner.js \
  --cwd "/Users/alanman/Developer/my-active-project" \
  --include-repo-context \
  --dont-ask --trace-level summary \
  "Do a repo health check: 1) git_status — flag uncommitted or unexpected files; 2) scan for new TODO/FIXME markers; 3) spot-check that README commands still match package.json scripts; 4) delegate a quick consistency pass to a 'verify' child via spawn_agent if anything looks off. End with a 5-line digest: OK items first, then warnings."
Why it fits the runner: this exercises the orchestration layer — the main loop can spawn_agent a focused verify child with its own context window for deeper checks, and --trace-level summary writes a local flight-recorder audit trail (~/.bridge-runner/traces/) so unattended scheduled runs stay inspectable. Since --dont-ask makes spawn_agent confirmations non-interactive and the tool surface stays read-only, an automation firing while you're away can't change anything.

Shared setup notes

  1. Every shortcut follows the screenshot pattern: Receive → Run Shell Script (zsh, Pass Input: as arguments) → Stop and Output / Show Result.
  2. The VS Code window hosting the Claude Local Bridge extension must be running (bridge on http://localhost:11437); no ANTHROPIC_API_KEY is needed — the bridge injects Claude Code OAuth from the macOS Keychain.
  3. Shortcuts' shell is non-interactive: always pass --dont-ask so confirmation prompts fail closed instead of hanging. Reserve --accept-edits for the worktree shortcut only, and avoid --allow-shell in unattended shortcuts.
  4. If Shortcuts' minimal PATH can't find node, use the absolute path (e.g. /opt/homebrew/bin/node).
  5. If caller auth is enabled on the bridge, add --caller-token or set BRIDGE_CALLER_TOKEN in the script.
