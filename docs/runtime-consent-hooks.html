10 At-Runtime Hooks for Consent, Boundaries & Project Scope

Bridge Runner — runtime performance review & hook proposals · grounded in src/runner/hooks/, tool-pipeline.js, permissions.js, shell-policy.js, budget-tracker.js, workspace-trust.js, session-health.js

Part 1 — Runtime performance findings (current hook path)

The runner already has a real hook substrate: HookDispatcher supports six events (session_start, pre_model_request, post_model_response, pre_tool, post_tool, session_end), loads .bridge-runner/hooks.json once at construction, is double-gated on workspace trust (trust.json fingerprint and the config's own trusted flag), and hook-runner.js executes commands through the same scanShellCommand policy, a scrubbed env (safety.buildSafeEnv()), and secret-scrubbed, 8 KB-capped output. That's a strong safety baseline. The performance picture is where care is needed:

P1 — Hooks execute with spawnSync. Every exec hook blocks the Node event loop for its full duration, with a default timeout of 120,000 ms. A slow pre_tool hook stalls the entire turn — and because the tool pipeline runs read-only tools as a parallel batch but fires sinks per-tool in order, a sync exec hook on pre_tool/post_tool re-serializes the batch and erases the parallelism win.
P2 — Hot-path events multiply cost. pre_tool/post_tool fire once per tool use. A 30-step session with ~4 tools/turn and one 50 ms exec hook on each event adds ~12 s of pure process-spawn overhead. Per-event hook matching is a linear filter over the config (fine at this scale), but subprocess spawn cost (~10–50 ms each on macOS) dominates.
P3 — Hooks observe but cannot decide. Dispatch results are logged (this.log) and recorded in the sink order (ledger intent → pre_tool hook → … → execute), but nothing consumes a hook's verdict to block an action. All ten proposals below need one shared change: a decision protocol (exit code 0 = allow, 2 = deny, stdout JSON = structured feedback) that tool-pipeline.js consults before execute. That keeps the effect-pairing invariant intact — a hook-denied tool still emits its paired tool_effect_result as an is_error result.
P4 — Payloads travel by convention, not contract. Hooks receive no structured input (payload isn't passed to the child process at all — only used for the log entry). Recommendation: pass the JSON payload on stdin, never argv/env, so paths and tool args can't leak into process listings or trip shell parsing.

Cross-cutting recommendations before adding any new hooks:

  • Switch executeHookCommand to async spawn with per-event default timeouts: 2–5 s for pre_tool/pre_model_request (they gate the loop), keep long budgets only for session_end. Time out ⇒ fail-open for observers, fail-closed for guards (explicit on_timeout field).
  • Add an in-process hook type ("action": "builtin" referencing vetted JS modules under src/runner/hooks/builtin/) for hot-path guards — pure JS checks cost microseconds, not milliseconds. Reserve exec for cold events.
  • Cache per-event hook lists at construction (Map<event, hooks[]>) and short-circuit dispatch when the event has zero hooks — makes the no-hooks case truly free.

1

Path-scope guard

pre_toolboundary

What: Independently re-verify that every path argument on a write/shell tool canonicalizes (realpath) to inside the workspace — and outside HARD_DENY_PATH_SEGMENTS (.git/, .ssh/, .aws/, .bridge-runner/…) — immediately before execution.

Why

permissions.js checks paths at proposal time, but a TOCTOU gap exists: a symlink created between check and execute (e.g., by a parallel worktree, a background shell, or an earlier tool in the same batch) can redirect a write. A last-instant re-check in the pipeline is the classic defense-in-depth answer, and it's the single highest-value guard because every other boundary depends on path confinement.

Implementation
  • Builtin module hooks/builtin/path-scope-guard.js — reuse safety.confinePath(ctx, p) (already used by _groupDisjointWrites) plus the blocked-segment list from shell-policy.js. Cost: one fs.realpathSync per path.
  • Wire the decision protocol (P3): in tool-pipeline.js, between ledger intent and execute, if any pre_tool hook returns {decision: "deny", reason}, fabricate an is_error tool result exactly like the existing permission-denial path — the effect-pairing invariant stays untouched.
  • Fail-closed: if realpath throws (dangling symlink), deny.

Consent-mode escalation sentinel

pre_toolconsent

What: Detect when the effective behavior of a session drifts above what the user's chosen permission mode implies — e.g., in acceptEdits mode, N consecutive auto-approved writes touching M distinct files within one turn triggers a one-time re-confirmation.

Why

The MODES matrix in permissions.js is static per session, but consent is contextual: a user who accepted "edit files without asking" for a rename refactor did not necessarily consent to 40 files rewritten in one turn. Rate-based re-consent converts a binary mode into a graduated one, catching runaway behavior (and prompt-injection-driven bulk edits) without nagging on normal work.

Implementation
  • Builtin with per-session counters (files-written set, writes-per-turn), thresholds in hooks.json ({"max_writes_per_turn": 10, "max_distinct_files": 25}).
  • On breach, return {decision: "ask", reason} — route through the existing confirm port (confirm.ask(proposedAction, timeoutMs)) that the pipeline already owns, so UX and timeout semantics match normal confirmations. One approval re-arms the counter at 2× the threshold.
  • Persist counters in the session store so --resume can't reset the odometer.
3

Network-egress re-consent

pre_toolconsent

What: Any shell command matching NETWORK_PATTERNS (curl, ssh, git push, npm publish…) requires explicit interactive consent even in dontAsk mode, where shell is currently allow.

Why

Egress is the one category where a mistake can't be undone with undo — data leaves the machine, pushes land on remotes, packages publish. shell-policy.js already detects these patterns; this hook upgrades detection into a consent checkpoint. It's also the strongest single mitigation against prompt injection from untrusted repo content, since exfiltration almost always needs egress.

Implementation
  • Builtin that calls the existing scanShellCommand and returns {decision: "ask"} on any network-classified issue — zero new pattern code to maintain.
  • Support a session-scoped allowlist ("allow git push origin feature-x for this session") stored in memory only — never persisted, so consent expires with the session.
  • In non-interactive runs (child workers, --dont-ask with no TTY), "ask" degrades to deny with a clear message — fail-closed, matching how ask_user_question is already unavailable to workers.

Blast-radius diff budget

post_toolscope

What: Accumulate a session-level change envelope — files touched, lines added/removed, files created vs modified — and warn at a soft cap / pause for consent at a hard cap.

Why

budget-tracker.js budgets tokens, wall time, and spawns — but not change surface, which is what the user actually experiences as "the agent went too far." A task pitched as "fix the failing test" that has modified 30 files is out of scope regardless of token spend. Mirroring the tracker's proven 80% soft-warning / hard-stop pattern makes this instantly familiar.

Implementation
  • Builtin on post_tool for write-category tools; edit_file/write_file results already know the path and content size — add a small {path, linesDelta, created} field to the hook payload from the pipeline.
  • Reuse the tracker structure: createChangeBudget({maxFiles, maxLines}) emitting the same structured warning shape at 80%, and a new STOP_REASONS.CHANGE_BUDGET_EXCEEDED at 100% that the kernel treats like CONTEXT_BUDGET_EXCEEDED (stop at loop boundary, resumable after consent).
  • Feed breaches into session-health.js DEGRADED_STOP_REASONS so resume guardrails recommend a fresh session after a blast-radius stop.

Outbound secret-leak scrubber

pre_model_requestboundary

What: Scan the outgoing message array for high-entropy tokens and known credential shapes (AKIA…, ghp_…, sk-…, PEM headers, JWT triplets) before the request leaves for the bridge; redact-and-warn or block on hits.

Why

Inbound defenses are solid (blocked basenames in permissions.js, scrubSecrets on hook output, safe env) — but they guard tool results. A secret can still enter context via a legitimately readable file (a fixture, a log, a config committed by mistake) and then travel to the API on every subsequent request. The pre-request boundary is the last exit door, and the only place that catches all sources at once.

Implementation
  • Builtin (never exec — you'd be piping the very secrets to a subprocess). Reuse/extend safety.scrubSecrets; add a Shannon-entropy check for unlabeled tokens ≥ 20 chars.
  • Performance discipline: scan only the messages appended since the previous request (keep a high-water mark), not the whole context each turn — turns an O(context) cost into O(delta).
  • Default action: redact to [REDACTED:sha256-prefix] and log; "on_hit": "block" config for stricter setups. Record every redaction in the ledger for the session-end audit (hook #10).

Lane & branch verifier

session_startscope

What: At session start, assert the runner is in the expected lane: cwd realpath matches the trusted fingerprint, git branch matches an expected pattern, remote URL matches the project, and no unexpected dirty state exists. Mismatch ⇒ refuse to start (or demand explicit override).

Why

Workspace trust (workspace-trust.js) answers "is this directory trusted?" — but not "is this the directory the user thinks they're in?" With worktree slots, multiple parallel runs, and resumable sessions, the most likely boundary failure isn't malice — it's a session resumed in the wrong worktree or on main instead of a feature branch. Cheapest possible insurance: it runs once per session.

Implementation
  • This one fits the existing exec action as-is (cold path, sync cost acceptable): a repo-committed script checked against hooks.json expectations ({"expect_branch": "^(feat|fix)/", "expect_remote": "github.com/org/repo"}).
  • Make session_start the first blocking-capable event: a non-zero exit prevents the kernel from entering the loop, before any model request is made.
  • Compare against workspace-fingerprint.js output to also catch "trusted path, but the tree was swapped underneath" (fingerprint drift since trust was recorded).

Scope-drift detector

post_model_responsescope

What: After each model response, compare the proposed tool uses against a declared scope manifest (glob allowlist derived from the task, e.g. src/runner/** + docs/**) and the live task checklist. First proposal outside scope ⇒ inject a steering note; repeated drift ⇒ pause.

Why

Boundary violations get blocked (hook #1), but drift is subtler: everything is individually permitted, yet the sum wanders off-task — the classic "went to fix a test, refactored the config system." The runner already has the machinery mindset for this (repeat-tool-detector.js injects steering notes; loop-autopsy.js detects semantic cycles); scope drift is the same pattern with a different predicate. Steering before execution is far cheaper than undoing after.

Implementation
  • Builtin between "model responded" and "pipeline executes": pure glob matching (picomatch) on proposed paths — microseconds.
  • Scope manifest sources, in priority order: explicit --scope flag → coordinator spec (coordinator-spec-compiler.js) → inferred from the first turn's task statement (loosest, warn-only).
  • Escalation ladder mirroring formatRepeatWarningNote: 1st drift → appended note ("path X is outside declared scope Y — confirm or refocus"); 3rd → ask via confirm port; never silently auto-deny, since scope inference can be wrong.
8

New-territory write confirmation

pre_toolscopeconsent

What: The first write into any top-level directory not yet touched this session (and any file creation, vs modification, outside already-visited directories) requires a one-time confirmation: "expanding into scripts/ — allow for this session?"

Why

Users think in territories, not files. "Yes, edit src/runner/" is a natural consent unit; per-file asks are noise and per-session acceptEdits is too coarse. This gives acceptEdits mode a memory: consent accumulates per directory as the session progresses, which is exactly how a human pair-programmer negotiates scope. It also makes hook #4's blast radius legible — the audit shows which territories were approved and when.

Implementation
  • Builtin with a session Set of approved top-level dirs (seed it from paths the user mentioned in their prompt — pre-consented territory).
  • Depth is configurable ("granularity": 1 for top-level, 2 for src/runner-level); default 1 to keep prompts rare.
  • Approvals recorded in the ledger as consent_scope_expanded events — survives resume, feeds hook #10's audit, and gives crash recovery an accurate consent picture.

Budget checkpoint consent

post_model_responseconsent

What: Upgrade the budget tracker's soft-cap warning (80% of tokens / wall / spawns) from a one-shot log line into an interactive checkpoint: pause at the loop boundary and ask "80% of budget consumed, task ~60% done — continue, wrap up, or stop?"

Why

Budget is consent expressed in resources. budget-tracker.js already computes soft caps and even threads parentRemaining through to child agents — but the warning is fire-and-forget (warned.input = true) and the user only regains control at the hard stop, when the remaining 20% is gone. A checkpoint converts "your money is spent" into "how do you want to spend the rest?", and pairing it with the session-health picture lets the user cut losses on a degraded run early.

Implementation
  • Builtin subscribing to the tracker's existing soft-cap emission (the warned flip is the natural trigger point — emit an event instead of only a warning object).
  • Three-way answer via ask_user_question-style options: continue (re-arm at 90%), wrap up (inject "finish current item, then summarize and stop" as a steering note), stop now (clean STOP_REASONS stop, resumable).
  • Non-interactive sessions: default to wrap up — the graceful-degradation choice — and log the auto-decision.
10

Consent audit trail

session_endconsent

What: At session end, compile a human-readable consent ledger: every ask/allow/deny with its reason, every hook decision (including #2/#3/#8 escalations), territories approved, redactions performed (#5), budget checkpoints (#9), and the final change envelope (#4) — written to .bridge-runner/runs/<run-id>/consent-audit.md.

Why

Consent without auditability is just optimism. All the raw data already exists — the session ledger's effect pairs, HookDispatcher.getLog(), the human log — but scattered across sinks in machine shape. A single post-session artifact closes the loop: the user can verify the run stayed inside what they approved, diff the outcome against expectations, and (crucially) tune hooks #1–#9 based on real false-positive/negative rates rather than guesswork. It's also the document you want when a run went wrong.

Implementation
  • exec action is fine here (cold path, session is over); or a builtin reducer over the ledger JSONL — prefer the builtin so it works even in untrusted-exec configurations.
  • Group by consent category: auto-allowed by mode / explicitly approved / hook-escalated then approved / denied. The interesting section is the third — that's where mode settings and reality disagree.
  • End with a one-line verdict the human-log can surface: “Stayed in scope: 47 actions, 2 escalations (both approved), 0 denials, 0 redactions.” Feed a machine-readable twin (consent-audit.json) to golden-eval.js so consent regressions become testable.

Part 3 — Suggested build order

  • Phase 0 (enabler): the decision protocol (P3), stdin payloads (P4), async spawn with per-event timeouts (P1), and the builtin hook type (P2). Every hook above depends on at least one of these.
  • Phase 1 (hard boundaries): #1 path-scope guard, #5 secret scrubber, #6 lane verifier — fail-closed, no UX cost, highest damage prevented per line of code.
  • Phase 2 (graduated consent): #3 egress re-consent, #8 new-territory confirmation, #2 escalation sentinel — these need the confirm-port routing.
  • Phase 3 (scope & stewardship): #4 diff budget, #7 drift detector, #9 budget checkpoint, #10 audit trail — quality-of-life layer that makes the first two phases tunable.
