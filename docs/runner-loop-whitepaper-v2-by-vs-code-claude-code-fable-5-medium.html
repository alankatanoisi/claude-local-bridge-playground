The Claude Local Bridge Runner:
An Architectural Analysis of a Bounded, Crash-Recoverable Agentic Execution Loop

Technical White Paper — Second Edition

Claude Local Bridge Playground · Runner Layer · Derived from direct source inspection of src/runner/run.js, src/runner/tool-pipeline.js, src/runner/tool-registry.js, src/runner/permissions.js, src/runner/safety.js, src/runner/model-client.js, src/runner/context-builder.js, src/runner/context-compactor.js, src/runner/session-ledger.js, src/runner/tool-catalog.js, and src/runner/kernel/contract.js

Abstract

We present a detailed architectural analysis of the Claude Local Bridge Runner, a local coding-agent harness written in plain Node.js (no framework, no SDK dependency) that drives the canonical agentic contract — prompt → model → tool_use → local execution → tool_result → repeat — against an Anthropic Messages endpoint served by a companion VS Code extension (“the bridge”) on localhost:11437. The system's central architectural commitment is a strict separation of loop-level policy from effect-level mechanism: a single bounded for loop in run.js owns budgets, compaction, prompt-cache placement, and termination, while a deep module, the tool pipeline, owns everything between “the model emitted tool_use blocks” and “completed, recorded tool results exist.” The two layers communicate through exactly one seam, the midTurnCheck callback. We formalize the invariants the implementation maintains — effect pairing (write-ahead intent/result journaling around every side effect), a two-phase read-parallel/write-serial execution schedule, plan-mode write fabrication, hard-deny survival under forced execution, and a strict prompt-cache breakpoint budget — and analyze the termination taxonomy of twenty-one machine-readable stop reasons. We conclude with a discussion of the design's trade-offs relative to hosted agent frameworks.

1. Introduction

The agentic contract at the heart of every modern coding assistant is deceptively simple. A language model receives a conversation history and a catalog of tool definitions. It replies with either prose (a final answer) or one or more tool_use content blocks (requests for the harness to act on its behalf). The harness executes the requested tools, appends the results as tool_result blocks in a new user message, and calls the model again. The loop terminates when the model replies without tool requests.

What separates a toy loop from a production harness is everything around that contract:

  • Bounded execution. Steps, tokens, dollars, and wall-clock time must all be capped, and each cap must produce a distinct, machine-readable outcome.
  • Crash recovery. Side effects (file writes, shell commands) must be journaled before they occur, so an interrupted run can be diagnosed and resumed.
  • Context management. Long tasks outgrow the model's context window; history must be compacted without destroying the model's working memory.
  • Permissioned execution. Writes and shell access require a policy lattice, human confirmation flows, and a class of denials that no flag can override.
  • Prompt-cache economics. Re-sending an ever-growing prefix every turn is ruinously expensive unless cache breakpoints are placed deliberately.

The Claude Local Bridge Runner solves each of these in roughly 5,000 lines of dependency-light JavaScript. This paper reconstructs the design from the source, bottom-up, with particular attention to the main loop in src/runner/run.js (§5) and the tool pipeline in src/runner/tool-pipeline.js (§6).

2. System Architecture

The project comprises two strata with a deliberately hard boundary:

LayerOwnsKey modules
Bridge (transport) OAuth credential access, keychain, request interception, upstream proxying to Anthropic. Exposes POST /v1/messages on 127.0.0.1:11437. src/server.js, src/proxy.js, src/credentials.js
Runner (agent) The agent loop, tool execution, permissions, context assembly, compaction, sessions, transcripts, archives, budgets, and the CLI. bin/local-bridge-runner.js, src/runner/**

The runner never touches credentials. It speaks plain, unauthenticated HTTP to loopback; the bridge injects the Claude Code OAuth Bearer token before forwarding upstream. This means the entire agent layer can be developed, traced, and tested as if the model were a local service, while token custody stays in one audited place.

2.1 The policy/mechanism split

Within the runner, responsibility divides again. The loop in run.js is policy: it decides whether the run may continue, what the request looks like, and how the run ends. The tool pipeline is mechanism: it decides how a tool_use block becomes a recorded tool_result, including permission checks, confirmations, execution scheduling, failure accounting, and journaling. The pipeline is constructed once per run (run.js:606) with its sinks and confirmation port fixed for the run's lifetime, and the loop's only mid-execution influence over it is the midTurnCheck callback (run.js:1221–1251), examined in §6.3.

This is the “deep module” pattern in Ousterhout's sense: a small interface (toolDefinitions(), executeTurn(), failureStreak, recordExternalFailure()) concealing substantial machinery. Notably, toolDefinitions() derives the tools shown to the model from the same ctx object that later gates execution — so the model can never see a tool the pipeline would refuse to run (tool-pipeline.js:22–27).

3. The Transport Layer

src/runner/model-client.js is a 262-line HTTP client with two entry points:

  • post(body, bridgeUrl, opts) — buffer the full JSON response;
  • postStream(body, cb, bridgeUrl, opts) — consume Server-Sent Events and reassemble the message incrementally.

Both share a keep-alive http.Agent with maxSockets: 1 (model-client.js:20), so every turn of a run reuses one TCP connection to loopback rather than paying a handshake per request. Both carry a 120-second timeout and attach a small _localBridge metadata object (status code, request id, Anthropic rate-limit headers) to the parsed response for the trace layer.

3.1 SSE reassembly

The streaming path is the most intricate part of the client. SSE frames arrive split arbitrarily across TCP chunks, so the client maintains a string buffer and splits on the frame boundary \n\n, retaining the trailing incomplete fragment (model-client.js:137–139). Within each frame it reconstructs the Anthropic streaming state machine:

  • message_start / message_delta accumulate usage counters;
  • content_block_start seeds fullContent[index], and for tool_use blocks initializes a per-index JSON accumulation buffer;
  • text_delta appends text (and echoes it live to stdout when streamOutput is set);
  • thinking_delta and signature_delta accumulate extended-thinking blocks — the signature is preserved verbatim because Fable/Mythos/Opus adaptive thinking requires it for multi-turn tool loops (model-client.js:184–188);
  • input_json_delta concatenates partial JSON for tool inputs, which is parsed only at content_block_stop, when the fragment is guaranteed complete.

The resolved value of postStream is shaped identically to the buffered response of post, so the agent loop is agnostic to whether streaming was used.

4. Run Lifecycle: From CLI Invocation to First Request

run(options) (run.js:328) performs a long, ordered initialization before the loop starts. The order matters; each stage feeds the next.

  1. Profile expansion. An agent profile and/or tool profile rewrites the options object (applyProfileToRunOptions, applyToolProfileToRunOptions), then the context policy is resolved and frozen into options.contextPolicy.
  2. Context construction. A mutable ctx object is built (run.js:402–422) holding the run's capability flags (acceptEdits, dontAsk, allowShell, plan, noNetwork), the working directory, undo log, task list, and spawn depth. ctx is the identity object of the run: permission caches, realpath caches, and LSP sessions are all keyed on it via WeakMap, so they are garbage-collected with the run.
  3. Bootstrap and trust gate. Unless explicitly skipped, runBootstrap validates the working directory, evaluates workspace trust, and loads instruction memory (CLAUDE.md/AGENTS.md-style docs, hashed for later delta detection). A blocked bootstrap terminates the run before any model call with stopReason: cwd_invalid or workspace_not_trusted.
  4. Fork-depth check. assertForkAllowed(ctx.spawnDepth) prevents runaway recursive agent spawning; the depth is inherited through the BRIDGE_RUNNER_SPAWN_DEPTH environment variable so it survives process boundaries.
  5. Durable infrastructure. The run id, session store, append-only ledger, hook dispatcher, transcript, human log, JSONL flight recorder (trace), archive collector, and budget tracker are all constructed (run.js:512–600). A recovery manifest seed (ctx.runManifestMeta) is prepared so undo last-run works even when session persistence is disabled.
  6. Pipeline construction. createToolPipeline({ctx, runId, confirm, sinks, …}) binds the pipeline to its sinks for the lifetime of the run (run.js:606–613).
  7. Message seeding. Either (a) a session is resumed — messages are rehydrated from the session store, gated by assertResumeAllowed health checks, and the new prompt is appended — or (b) a fresh message list is created via buildUserMessage, optionally prefixed with a dynamic environment block (cwd, git HEAD, instruction hash) when the context policy excludes dynamic data from the system prompt (run.js:727–800). Notably, resuming from raw transcripts is deprecated and refused: the session store, not the transcript, is the source of truth (§10.1).
  8. Prompt assembly. resolveSystemPrompt (§8.1) and buildRepoContextBlock produce the system prompt and the optional session-stable repository context block; pipeline.toolDefinitions() produces the tool catalog.

Only after all of this does control enter the loop.

5. The Agent Loop in Detail

The loop itself is a single bounded for statement (run.js:869): 

for (let step = 1; step <= steps; step++) { … }   // steps = --max-steps, default 16

Each iteration — one turn — passes through nine phases in fixed order. We treat each in turn.

5.1 Phase 1 — Instruction-delta detection

Before building the request, the loop asks instruction-delta whether the project's instruction documents (CLAUDE.md and friends) changed since the snapshot taken at run start (run.js:872–890). Two outcomes:

  • Small diff → a delta block is pushed as a user message, telling the model what changed without invalidating the cached system-prompt prefix.
  • Large rewrite → instruction memory is reloaded and the system prompt is rebuilt wholesale, deliberately paying the cache invalidation because the old prompt is now misleading.

This is a cache-economics decision disguised as a correctness feature: system-prompt changes invalidate the entire prompt cache prefix, so the runner routes small edits through the message list (append-only, cache-preserving) and reserves prompt rebuilds for rewrites.

5.2 Phase 2 — The compaction ladder

applyCompactionLadder(messages, system, policy) (§8.2) may rewrite the message history to shed tokens. If any stage fired, a compaction event is emitted, the ledger records compaction_applied, and a monotonically increasing compactionGeneration counter is persisted to the session — downstream health checks use this to recommend a fresh session once a conversation has been squeezed too many times.

5.3 Phase 3 — Prompt-cache breakpoint placement

applyCacheControlBudget(system, tools, messages, repoContextBlock) (run.js:124–163) decorates the request with cache_control markers. Anthropic permits at most four breakpoints per request; the bridge consumes one for its prepended OAuth identity block, so the runner budgets three (BRIDGE_OAUTH_CACHE_RESERVE = 1):

#PlacementWhat it caches
1Last block of the system prompt (or the repo-context block, when present, which then becomes the only system marker)The entire system prefix
2The last tool definitionThe whole tools array
3Last block of the second-most-recent messageThe stable transcript prefix

The third placement embodies the key insight: the most recent message is always the freshly appended turn, which by definition was not in the previous request — marking it would create a cache entry that is immediately invalidated. Marking the second-most-recent message instead (markStableTranscriptPrefix, run.js:193–203) caches everything the previous turn already paid to process. All markers use a shared ttl: '1h' (RUNNER_CACHE_CONTROL, run.js:105) because Anthropic rejects mixed TTL ordering against the bridge's own 1-hour marker. The function never mutates its inputs (messages are shallow-cloned), and it counts the breakpoints it produced, throwing if the count ever exceeds the budget (run.js:151–160) — a loud failure preferred over silent server-side cache degradation.

5.4 Phase 4 — Request construction and dispatch

The request body (run.js:945–954) is a plain Anthropic Messages payload: model, max_tokens, the cache-decorated system/messages/tools, optional stream: true (only when output format is text), optional temperature, and optional output_config.effort (the Claude 5 effort control, validated against {low, medium, high, max}). It is posted with per-turn trace headers (x-local-bridge-trace-id, -run-id, -trace-turn, -trace-level) that let the bridge write a correlated trace file, so a single run id can be followed across both processes.

A bridge error does not immediately kill the run: pipeline.recordExternalFailure() advances the same consecutive-failure streak used for tool failures, and the loop continues to retry the turn until the shared streak reaches MAX_CONSECUTIVE_FAILURES = 2 (run.js:991–998). Unifying the two failure sources in one counter is a deliberate simplification: from the run's perspective, “the world is failing repeatedly” matters more than which component failed.

5.5 Phase 5 — Response accounting and budget boundary

Usage is folded into a running total (addUsage, including cache read/creation tokens), and emitBudgetBoundary consults the budget tracker for input/output token budgets (including budgets inherited from a parent agent via BRIDGE_RUNNER_BUDGET_*_REMAINING environment variables — a child agent spends from its parent's allowance). A tripped budget terminates the run right here, before any tool from this turn executes. A separate, cruder guard stops the run when cumulative tokens exceed twice --max-context-tokens (run.js:1068–1104), a runaway-cost backstop with an explicit user-facing remediation message.

5.6 Phase 6 — Termination test

The assistant message is appended to history and the session is persisted before tools run (run.js:1119–1125) — so a crash during tool execution loses no conversational state. Then the loop extracts tool_use blocks. If there are none, the model has answered: the run finalizes with stopReason: success, writes final text to every sink, builds a post-mortem “autopsy” (tool histogram, steps, usage), and returns (run.js:1179–1219). A per-turn cap (--max-tool-calls-per-turn) guards against a pathological response containing hundreds of tool calls.

5.7 Phase 7 — Tool execution

pipeline.executeTurn(step, toolUses, {midTurnCheck}) executes the turn's tools (§6) and returns {toolResults, outcomes, failureStreak, escalated, aborted}.

5.8 Phase 8 — Mid-turn policy checkpoint

The loop's midTurnCheck closure (run.js:1224–1250) is invoked by the pipeline at exactly one point: after the read batch has completed and been recorded, before any write executes. It evaluates, in order: semantic cycle detection (has the model been issuing the same tool call signature repeatedly? — detectSemanticCycles over the accumulated tool history), the wall-clock budget, the cost budget (estimated dollars from the pricing table), and the token budget tracker. Any verdict aborts the turn with a typed stop reason. The placement is the whole point: stopping between reads and writes means a runaway run is halted at a moment when it has observed the world but not yet mutated it.

5.9 Phase 9 — Result append and next turn

The pipeline's toolResults array becomes a single user message (run.js:1282), the session is persisted again, and the loop iterates. If the for loop exhausts maxSteps without a text-only response, the run ends with stopReason: max_steps, an autopsy, and exit code 1.

6. The Tool Pipeline: A Deep Module

createToolPipeline (tool-pipeline.js:160) closes over the run context, the confirmation port, and seven nullable sinks. Its module header documents five invariants; we restate them and show how each is enforced.

6.1 Invariant: effect pairing

Every tool use appends a ledger tool_effect_intent event with a fresh effectId before execution, and exactly one tool_effect_result with the same id after — on success, failure, thrown error, permission denial, plan-mode fabrication, and user denial alike. This is write-ahead logging applied to agent side effects: after a crash, an intent without a matching result identifies precisely which effect may have partially occurred. The ledger surfaces these as getPendingIntents(), and the loop reports them at startup as “recovered from incomplete prior run (n pending effects)” (run.js:527–534).

The ledger is correspondingly the one critical sink: an append failure aborts the turn, because an effect that cannot be journaled would break the recovery guarantee. Every other sink (transcript, human log, trace, archive, hooks, stream events) is wrapped in bestEffort() — a broken log file must never alter a tool result (tool-pipeline.js:174–191).

6.2 Invariant: deterministic sink order

For each tool the fan-out order is fixed: ledger intent → pre_tool hook → tool_use event → trace → transcript → execute → transcript result → human log → trace → archive → ledger result → post_tool hook → tool_result event. Consumers of any single sink can therefore rely on a stable event grammar.

6.3 Two-phase execution: reads parallel, writes serial

executeTurn partitions the turn's tool uses by catalog category (CATEGORIES[name] === 'read-only' vs. everything else) and schedules them differently:

  • Read phase. All read-only tools run as one Promise.allSettled batch (executeReadOnlyBatch). If any sibling fails, every surviving result is annotated “some reads in this batch failed — this result may reflect stale state,” warning the model against reasoning from a partial snapshot.
  • Mid-turn checkpoint. The loop's policy callback runs (§5.8).
  • Write phase. Write, shell, recovery, and orchestration tools run serially, in the order the model emitted them, preserving the model's intended sequencing of dependent edits.

One optimization complicates the serial write phase. Under --accept-edits (and never in plan mode), consecutive writes whose canonicalized paths are pairwise disjoint — neither identical nor in a parent/child relation, tested with a separator-aware prefix check (_arePathsDisjoint, tool-pipeline.js:72–84) — are pre-executed in parallel via Promise.all. The serial loop then consumes the cached results, so sinks still observe events in model-emitted order even though the filesystem work overlapped. Paths are canonicalized through safety.confinePath first, so symlink aliasing cannot make two “different” paths collide (_groupDisjointWrites, tool-pipeline.js:86–114).

6.4 The confirmation flow and plan-mode fabrication

When the permission gate returns ask, the registry surfaces {needsConfirmation: true, proposedAction} instead of executing. The pipeline then:

  1. emits approval_required to the event stream and transcript;
  2. in plan mode, fabricates a synthetic success — "Plan mode: would <action>" — in exactly one place (fabricatePlanResult, tool-pipeline.js:283), so the model can continue planning multi-step work without any effect occurring;
  3. otherwise, awaits the confirmation port. On “allow,” the tool is re-executed through executeForce (which re-checks permissions with acceptEdits semantics but still honors hard denies, §7.2); on “deny,” the result becomes {ok: false, text: 'User denied this action.'} and flows back to the model like any other failure.

6.5 Failure-streak escalation

Failures on the write/shell path advance a streak counter; any success resets it. At the limit (2 consecutive), an escalation notice is appended to the tool result text itself before it is recorded, so the model, the transcript, and the human all see the same warning (noteWriteOutcome, tool-pipeline.js:290–325). Read failures are deliberately excluded from the streak — a failed grep is routine; a failed write twice in a row suggests the model is stuck.

6.6 Post-turn verification appendix

If any write succeeded (and not in plan mode), an optional test watcher runs the project's configured verification and appends its outcome to the last write's result text (tool-pipeline.js:494–506) — feeding immediate test feedback into the model's next turn without a separate message.

6.7 Repeat-tool detection

A per-run detector (repeat-tool-detector) watches for the same tool being called with the same arguments within a window; on trigger it appends a warning note to the result, tags the result envelope, and journals a repeat_tool_warning ledger event. This is the soft precursor to the loop-level semantic-cycle stop: first the model is warned inside its own observation stream; only persistent repetition halts the run.

7. The Permission Lattice and Safety Chokepoint

7.1 Mode × category matrix

Every tool belongs to exactly one category, declared in its own module and validated at load time by tool-catalog.js (a half-registered tool throws at require time, not at runtime). The active mode is derived from context flags, and the decision is a simple matrix lookup (permissions.js:13–54):

Mode \ Categoryread-onlywriteshellrecoveryorchestrationworktree
defaultallowaskaskallowaskask
acceptEditsallowallowaskallowaskallow
dontAskallowaskallowallowallowallow
acceptEditsAndDontAskallowallowallowallowallowallow
planplan_only (every category → dry-run fabrication)

Two asymmetries encode the project's safety rules directly: acceptEdits does not unlock shell, and dontAsk does not unlock writes — each flag relaxes one axis only. Shell additionally requires --allow-shell merely to be visible; the matrix is only consulted after that gate.

7.2 Severity: bypassable vs. hard denial

Every decision is enriched with metadata (category, mode, ruleId, matchedGuards, severity, and a human-readable explanation). The severity field creates two distinct classes of “no”:

  • hard_deny — path escapes of the working directory, the secret deny matrix (.env*, key files, credential JSON, .ssh, .aws, .claude, .git…), shell commands referencing blocked paths or credential environment variables, unknown tools, profile denials, and child-agent spawn attempts. These survive forced execution: executeForce re-checks permissions with maximally permissive flags but still refuses anything with severity: 'hard_deny' (tool-registry.js:229–235). No CLI flag, confirmation, or parallel pre-pass can cross this line.
  • bypassable_ask/bypassable_deny — mode-policy asks and the shell-hidden default, which flags and confirmations legitimately lift.

7.3 The safety chokepoint

safety.js is the single module through which all paths and all tool output pass:

  • confinePath resolves and realpath-checks containment inside --cwd (symlinks cannot escape);
  • validateCwd refuses to operate inside system directories (/, /etc, /usr, /tmp, …);
  • isPathBlockedByDenyMatrix applies the secret-pattern predicates to every resolved path, read or write;
  • scrubSecrets redacts API keys, tokens, and key blocks from every tool result before it reaches messages, transcripts, or events — streaming tools pass through a sliding-window scrubber (makeStreamingScrubber) so a secret split across two chunks is still caught, with a 10 MiB hard cap on assembled output;
  • buildSafeEnv hands shell subprocesses a filtered environment.

In tool-registry.runAndScrub, boundary auto-summarization of oversized results runs after scrubbing (tool-registry.js:133) — the ordering comment is explicit that summarization must not be able to smuggle bytes past the redactor. When redaction fired, a notice is prepended telling the model that quoted secret-like strings are not byte-exact.

7.4 Decision and realpath caching

Permission checks are hot-path (every tool call), so two WeakMap-keyed caches exist: a per-session realpath cache and a full permission-decision cache keyed on (tool, flags, profile, canonical args) with LRU eviction at 512 entries. Two rules keep the caches sound: ask decisions are never memoized (the human must always be reached), and every successful write invalidates cached entries for the touched path in both caches plus the search-result cache (tool-registry.js:203–214).

8. Context Engineering: Compaction and Cache Placement

8.1 System-prompt assembly

The default system prompt is deliberately minimal: an intro (“you are a minimal coding agent…”), a tool summary, four rules, then opt-in sections — instruction docs, auto-memory, a skills index — each gated by the context policy and passed through applyContextBudget, which caps the labeled sections against a token budget (context-builder.js:60–97). Overrides layer in a fixed precedence: CLI --system-prompt > --system-prompt-file > ~/.bridge-runner/SYSTEM.md then <project>/.bridge-runner/SYSTEM.md (project wins) > the built prompt; append files and flags concatenate afterwards (system-prompt.js:45–80). Dynamic per-machine facts (cwd, git HEAD, instruction hash) can be excluded from the system prompt entirely and moved into the first user message, keeping the system prefix byte-identical across machines and sessions — again a cache-stability decision.

8.2 The compaction ladder

context-compactor.js implements loss functions ordered cheapest-first, applied only as token pressure (estimated at chars/4) demands:

StageActionLoss profile
clipTruncate any single tool_result over 12,000 chars, appending a marker inviting the model to re-fetch with read_fileBounded, recoverable
snipRemove bodies of old tool_results beyond a message-count threshold, preserving the most recent turns verbatimOld observations
cost/ghostInsert “ghost” summary blocks that tell the model what was intentionally compressedMetadata replaces content
summarizeFull history summarizationHighest; last resort

Message-count-triggered stages are opt-in (snipOnMessageCount: false by default): the policy comment notes that a long task can have many small turns while remaining far under the token budget, and snipping those turns early forces the model to wastefully re-read files it has already inspected. Ghost blocks are a notable courtesy to the model: rather than silently deleting history, the compactor leaves an explicit record of what was removed, letting the model decide whether to re-derive it.

9. Termination: The Stop-Reason Taxonomy

Every run ends with exactly one stopReason from a frozen taxonomy of twenty-one values (kernel/contract.js). They group naturally:

Non-success reasons set process.exitCode = 1, and most build an autopsy — a structured post-mortem (stop reason, step count, tool-call histogram, usage) written next to the session file — that feeds session-health scoring, which in turn may recommend a fresh session on the next invocation. All terminal paths converge through a single completeRun() finalizer (run.js:638) that disposes LSP sessions, prints the usage/cost summary once, flushes the session store synchronously, closes the ledger fd, and finalizes the archive export — one site covering success, budget stops, and errors alike.

10. Persistence, Recovery, and Observability

10.1 Three durability layers

ArtifactPurposeFailure semantics
Session store (*.state.json)Source of truth for resume: full message list, runner state (tasks, undo log, failure streak, compaction generation, health)Debounced saves (saveSoon), synchronous flush at run end
Session ledger (*.ledger.jsonl + cursor sidecar)Append-only, monotonically sequenced event journal; effect intent/result pairing; crash forensicsCritical: append failure aborts the turn. Kept-open fd (openSync/writeSync); the cursor sidecar records {seq, offset, pendingIntents} so resume avoids a full file scan, falling back to a scan if the cursor is missing, version-mismatched, or ahead of the file (corruption)
Recovery manifest (.bridge-runner/runs/)Per-run undo log powering undo last-runBest-effort; written at every exit branch, deliberately even when session persistence is disabled, because the user still owns the files the run touched

10.2 Observability sinks

Beyond durability, five best-effort surfaces observe the same event stream: the JSONL transcript (machine replay), the human log (prose narration), the JSONL flight-recorder trace (per-boundary payload capture, correlated with the bridge's own trace by run id and explicit visibility_notes declaring which facts only the runner or only the bridge can see), the archive collector (a browsable bundle of the entire run), and the stream-json stdout protocol (a stable event grammar — system, model_request, assistant, tool_use, tool_result, approval_required, compaction, budget, error, result — for programmatic consumers). Every one of these passes through the secret scrubber; redaction is a property of the boundary, not of any individual sink.

10.3 Hooks

A HookDispatcher fires project-local hooks at session_start, pre_model_request, post_model_response, pre_tool, post_tool, and session_end, and additionally mirrors every ledger event. Hooks are gated on workspace trust and are best-effort by construction — extension code cannot corrupt the run.

11. Discussion

Three properties distinguish this design from the typical framework-mediated agent loop.

The single-seam architecture. Most harnesses interleave policy checks throughout execution; here the loop's entire mid-execution authority is one callback invoked at one point (post-read, pre-write). This yields an unusually clean testing story — the pipeline can be exercised with a stub midTurnCheck, and the loop with a stub pipeline — and a precise safety statement: a stop verdict can never leave a half-applied write behind, because no write has started when the verdict is taken. The cost is granularity: a wall-clock overrun during a long serial write phase is only detected on the next turn.

Cache economics as a first-class design force. The breakpoint budget arithmetic (4 minus 1 reserved for the bridge), the stable-prefix marking rule, the routing of instruction deltas through user messages, the frozen tool ordering in the catalog (“insertion order is the order tools are offered to the model… so the tools array and its prompt cache is stable”), and the relocation of machine-specific facts out of the system prompt are all expressions of a single principle: the request prefix is an asset with a balance sheet. The loud throw-on-overbudget check turns a silent economic regression into an immediate test failure.

Defense in depth with an explicit floor. The permission system is not one gate but a stack: tool visibility (the model cannot request what it cannot see), the category×mode matrix, profile constraints, the confirmation port, and finally the hard-deny floor that survives even executeForce. The floor is small and non-negotiable — path confinement, the secret deny matrix, credential environment variables — and everything above it is configurable. The same layering appears on the output side: scrub, then summarize, then annotate, in an order chosen so later stages cannot undo earlier guarantees.

Limitations follow from the same choices. Token estimation is heuristic (chars/4), so compaction and predictive budgets are approximate. The disjoint-write parallelism only accelerates --accept-edits runs. And the semantic-cycle detector operates on tool-call signatures, not on semantic progress, so a model rephrasing the same futile plan with varying arguments evades it until a resource budget intervenes. These are conscious trades: each keeps the mechanism inspectable by a single reader, which — for a personal research harness whose second maintainer is a language model — is itself a load-bearing requirement.

The runner loop, in brief (the paper covers all of this in depth)
The whole agent is one bounded for loop in run.js:869 (default 16 steps). Each turn: detect instruction-file changes → run the compaction ladder → place prompt-cache breakpoints → POST to the bridge at localhost:11437/v1/messages → account usage against budgets → if the model replied with no tool_use blocks, stop with success → otherwise hand the tool calls to the pipeline → append results as a user message → repeat.

The three ideas I found most architecturally interesting:

One seam between policy and mechanism. The loop never executes tools; tool-pipeline.js does, and the loop's only mid-execution authority is the midTurnCheck callback, invoked at exactly one point — after all parallel reads finish, before any serial write starts. So a cycle/cost/wall-clock stop can never leave a half-applied write behind.
Effect pairing is write-ahead logging. Every tool call journals a tool_effect_intent before executing and a matching tool_effect_result after — even on denial, error, or plan-mode fabrication. The ledger is the only critical sink (its failure aborts the turn); transcript, trace, human log, archive, and hooks are all best-effort.
Prompt-cache economics shape the code everywhere. Anthropic allows 4 cache_control breakpoints; the runner uses 3 and reserves 1 for the bridge's OAuth identity block, marks the second-most-recent message (the newest one would immediately invalidate its own cache entry), routes small CLAUDE.md edits through user messages to avoid rebuilding the cached system prompt, and throws loudly if the breakpoint count is ever exceeded.
The paper also documents the permission matrix (why --accept-edits doesn't unlock shell and --dont-ask doesn't unlock writes), the hard-deny floor that survives even forced execution, the secret-scrubbing chokepoint, the compaction ladder (clip → snip → ghost → summarize), all 21 stop reasons, and the three durability layers (session store, ledger + cursor sidecar, recovery manifest).
