⏺ Vuln 1: Sensitive Data Exposure: src/handlers/anthropic.js:18-32

  - Severity: Medium
  - Category: sensitive_data_exposure
  - Description: When logRequests is enabled, the new dumpCapture path writes
  the full incoming /v1/messages request body verbatim to the VS Code Output
  channel (and thus to VS Code's on-disk log files). Headers are redacted, but
  the body — which contains user prompts and any content pasted into them (file
  contents, credentials, tokens, PII) — is not truncated or redacted. The
  pre-existing verboseLog capped output at 300 chars; this new code path does
  not.
  - Exploit Scenario: A user enables claudeLocalBridge.logRequests for
  troubleshooting and pastes a secret (API key, private file) into a Claude
  prompt. The full prompt body is persisted to ~/Library/Application 
  Support/Code/logs/..., where any other process, extension, or backup tool with
   read access to the user profile can later read it. Logs may also be shared
  when the user attaches them to a bug report.
  - Recommendation: Truncate the body to a small length (e.g., 500–1000 chars)
  before logging, redact obvious secret patterns, or gate full-body dumping
  behind a separate explicit setting (e.g., claudeLocalBridge.dumpRequestBodies)
   distinct from logRequests.

  ★ Insight ─────────────────────────────────────
  - The fingerprint whitelist in src/fingerprint.js is well-designed:
  authorization, cookie, and x-api-key are excluded from capture, and the
  /v1/debug endpoint emits only Object.keys() of the live fingerprint — so OAuth
   tokens cannot leak through either the fingerprint store or the debug surface.
  - The 401 retry in the streaming OpenAI handler correctly drains the upstream
  response (upRes.resume()) and gates on a retry flag, avoiding both socket
  leaks and infinite retry loops — a common bug in streaming proxies.
  - One sub-finding (in-process header poisoning of the captured fingerprint by
  a sibling VS Code extension) was considered but scored at confidence 7 and
  filtered out per the ≥8 threshold; worth noting if the threat model ever
  expands to untrusted co-resident extensions.

❯ for this extension, how viable is this bridge?                                

⏺ Honest assessment of where this bridge stands technically and strategically.
  I'll walk through the viability dimensions one by one — the answer isn't a
  single number, it's "works today, brittle long-term, with one significant
  policy risk."

  1. Does the core mechanism actually work?

  Yes, and the design is more sophisticated than most bridges of this kind.
  Three things stand out from the code:

  - Live token interception (src/interceptors/https.js) patches https.request
  and fetch globally inside the VS Code extension host, capturing whatever
  credential Claude Code itself just minted. That sidesteps the hardest part of
  any "use my Pro subscription" bridge — token rotation. Most competing bridges
  stop working in 2–4 hours when the OAuth access token expires; this one
  auto-refreshes because Claude Code does the refresh and the bridge just reads
  the new token off the wire.
  - Adaptive fingerprint capture (src/fingerprint.js) is the right architectural
   move. Instead of hardcoding a claude-cli/2.1.119 user-agent and a fixed
  anthropic-beta list (which breaks every CLI release), it learns the live
  header set from the running Claude Code and replays it. That's why the file
  uses a strict whitelist excluding authorization/cookie/x-api-key — it's
  reading the exact wire format Anthropic's gateway currently expects.
  - Five-tier credential fallback means it degrades gracefully: live
  interception → env vars → Keychain → file → manual setting. If interception
  fails on a given OS or VS Code version, the older paths still produce a
  working token.

  Verdict on raw functionality: high. The OpenAI Chat Completions → Anthropic
  conversion in src/handlers/openai.js, the streaming pipe in src/proxy.js, and
  the 401 retry path are all thoughtfully done.

  2. How brittle is it to upstream changes?

  Moderately brittle, but self-healing for most changes.

  The fingerprint adapter is the key insurance policy here. When Anthropic adds
  a new required header or bumps the anthropic-beta list, the bridge picks it up
   automatically the next time Claude Code makes a request. That's a real
  engineering win — most "use Pro for API" projects on GitHub die within weeks
  because they hardcode headers.

  What it won't survive:
  - Anthropic switching to mTLS or attested-client signing. No replay attack
  works against an HMAC over the request body keyed by a per-client secret
  derived from a TPM/Secure Enclave. There's no public signal Anthropic is doing
   this, but it's the canonical defense against exactly this class of bridge.
  - A new required header that comes from a source other than Claude Code's 
  outgoing HTTP (e.g., derived from a websocket challenge or a code path the
  interceptor doesn't see).
  - Endpoint path changes like /v1/messages?beta=true becoming /v2/.... The path
   is captured live (messagesPathFor), so this is partially handled, but only
  after Claude Code itself starts using the new path.

  3. The policy / ToS dimension — the real risk

  This is where viability gets uncomfortable.

  Anthropic's Pro/Max subscription terms cover interactive use of Claude Code,
  not "expose my subscription as a local API for arbitrary tools." The bridge's
  marketing line ("any LLM tool can use your Pro subscription") is in tension
  with that. I'm not your lawyer, but several specific design choices increase
  the visibility of this:

  - The OAuth-mode requests (src/handlers/anthropic.js when creds.accessToken is
   present) replicate the full Claude Code identity — billing header, session
  ID, CLI user-agent, x-stainless-* SDK fingerprint, the system-prompt prefix
  blocks. From the gateway's perspective, the only thing distinguishing your
  bridge traffic from genuine CLI traffic is request patterns (volume,
  time-of-day, prompt shapes, lack of a tty).
  - That's effective at avoiding accidental rejection. It's also exactly the
  pattern fraud teams build heuristics for: "looks like CLI, behaves like a
  server." If/when Anthropic decides to enforce, the technical signal is
  detectable and the account-level remediation is cheap (rate-limit, downgrade,
  or ban).
  - The x-claude-code-session-id reuse across non-CLI tools (Cursor,
  Continue.dev, etc. as the README advertises) is a particularly distinctive
  signal because session IDs in real CLI use are fairly short-lived and tied to
  one process.

  So the technical viability is high, the account-risk viability is the soft 
  underbelly. For personal use under modest volume it's almost certainly fine
  today; for shared/team deployment or high QPS, the risk goes up sharply.

  4. Local security posture

  Reasonable for a personal-use tool. Two concrete observations from the diff
  just reviewed:

  - The fingerprint whitelist correctly excludes authorization, cookie, and
  x-api-key, so OAuth tokens don't leak into the fingerprint store or the
  /v1/debug endpoint. Good.
  - logRequests mode dumps full request bodies to the VS Code Output channel
  (the finding in the security review). For a tool that proxies subscription
  credentials, this is the kind of feature that should default to truncated
  dumps.

  CORS is restricted to localhost origins, the listener binds to localhost, and
  the bridge doesn't store credentials itself — it reads them on demand from
  existing sources. That's the right shape.

  5. How does it compare to alternatives?

  1. The OpenAI API spec is a de facto standard; the Anthropic one isn't. Years
